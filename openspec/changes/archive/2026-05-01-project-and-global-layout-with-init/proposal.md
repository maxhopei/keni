## Why

Step 02 landed the storage interfaces and file-backed adapters: `TicketStore`, `PRStore`, `ActivityLogStore`, and `ConfigStore` exist, write atomically to `.keni/`, and have in-memory test doubles backing every contract test. They are *capable* of producing the spec §5.1 / §5.2 layout, but no command has yet asked them to do so — running `ls .keni/` in a fresh project still returns nothing. This is the moment Keni gains a footprint on disk and a way to bootstrap projects: `keni init` runs in any folder, initialises git if needed, creates the `.keni/` directory tree exactly as `spec.md` §5.1 prescribes, generates a stable `project_id`, writes the initial `project.yaml` via `ConfigStore`, makes sure the user's `~/.keni/` global directory exists with a stub `config.yaml`, merges Keni's required entries into `.gitignore`, and stages an initial commit so the project is committable from the start. Per spec §7.1, this is step 2 of the user's getting-started flow ("Run `keni init` in an empty or existing folder"), and it unblocks every subsequent step that assumes a real `.keni/` exists — REST in step 04, MCP in step 06, scheduler in step 08, engineer runtime in step 09, and so on. Without this step, those layers would be testable only against in-memory doubles or hand-rolled fixtures; with it, every later step gets a real, idempotent, git-friendly project to run against.

## What Changes

- Introduce a new `keni init [path]` CLI subcommand in `@keni/cli` whose behaviour matches `spec.md` §7.1 exactly:
  - Resolves the target directory (defaulting to the current working directory) and verifies it is writable.
  - **Initialises git** (`git init`) if the target is not already a git repository. If it is already a repo, leaves the existing repo untouched.
  - **Creates the `.keni/` directory tree** matching `spec.md` §5.1 prototype scope: `tickets/`, `prs/`, and `activity/` directories created up front (empty), plus a `project.yaml` and a `state.json` skeleton at the top of `.keni/`. Spec / CR / chat directories are deliberately **not** created — they belong to MVP steps 14 and 15.
  - **Generates a stable `project_id`** — UUIDv4 — and writes the initial `project.yaml` via `ConfigStore.writeProjectConfig` so the write is atomic and goes through the storage abstraction (no raw `Deno.writeTextFile`). The initial config carries `project_id`, a `name` derived from the directory's basename, an `agents` roster with the default prototype agent (`alice`, role `engineer`), and the prototype default schedules. Stack is left unset for the prototype.
  - **Writes a `state.json` skeleton** containing a `watermarks` placeholder object — git-ignored, regenerated by later steps. `state.json` is the transient runtime-state file referenced by spec §5.1 and §6.2; this step lays down the empty shape, not the queue/checkpoint logic (those land in steps 08 and 14+).
  - **Ensures the user's `~/.keni/` directory exists** with a stub `config.yaml` if one is not already present (per spec §7.1: "On first use, Keni creates `~/.keni/` with an empty global `config.yaml`"). The stub is empty (`{}`) but well-formed YAML so future config-edit flows have a file to amend. The `~/.keni/logs/` directory is created up front; `~/.keni/workspaces/` is left for engineer step 09 to populate per agent.
  - **Merges Keni-required entries into `.gitignore`**: `.env`, `.keni/state.json`, build artefacts (`dist/`, `build/`), and `node_modules/`. If a `.gitignore` already exists, only missing entries are appended (no rewrites, no comment churn). If none exists, a fresh one is created with a leading comment block identifying it as Keni-managed.
  - **Stages and commits** the `.keni/` scaffold, the (possibly extended) `.gitignore`, and the initial commit message. Strategy: a single git commit covering all init-introduced files. Re-runs do not produce empty commits.
  - **Idempotent.** Re-running `keni init` on an already-initialised project is a no-op with a clear, structured message naming what already exists (e.g., "Project already initialised at /path/to/project (project_id: …); nothing to do."). Partial states (e.g., `.keni/project.yaml` exists but `tickets/` does not, because the user removed it) repair the missing pieces and continue, never overwriting an existing `project.yaml`.
- Extend `ConfigStore` with `writeGlobalConfig(config: GlobalConfig): Promise<void>` so `keni init`'s global-bootstrap path uses the same atomic-write contract as the project bootstrap. The existing read-only treatment of `~/.keni/config.yaml` (Decision 11 in the storage change's design.md, "the global file is created on install and edited manually in prototype") was a deferral; this step picks up that deferral. Both file-backed and in-memory `ConfigStore` adapters gain the method, the shared contract test gains a write-then-read-back scenario, and `mod.ts` re-exports nothing new (no new types — `GlobalConfig` was already exported in step 02). This is a delta on the `storage` capability, not a new capability.
- Layered config resolution: no functional change to `ConfigStore.resolve()`, which already overlays project on global per `spec.md` §5.2. This step makes `resolve()` actually meaningful in production by ensuring both layers exist on disk after `keni init`.
- A new `project-layout` capability spec captures the on-disk contract end-to-end: `.keni/` and `~/.keni/` shapes, `keni init` UX, idempotency semantics, the `.gitignore` merge policy, the initial-commit policy, the `project_id` generation rule, and the boundary between this capability and `storage` (which owns the file shapes and atomicity; this capability owns the bootstrap orchestration).
- Cross-cutting docs: a "Getting started" section is added (or expanded) in the root `README.md` showing the `keni init` invocation and the resulting layout. The CLI itself prints the documented post-init layout summary on success so the user sees what was created and where.
- Test coverage:
  - **Unit tests** for the new `ConfigStore.writeGlobalConfig` method (round-trip via shared contract test, file-adapter atomicity test).
  - **Integration tests** for `keni init` covering: fresh empty directory → valid project; existing git repo → preserves history, adds keni commit; existing non-empty non-repo directory → init creates repo and commits; existing `.gitignore` is merged not overwritten; existing initialised project → idempotent no-op; partial state (keni dir present, project.yaml missing) → repaired; the on-disk layout matches the documented golden layout byte-for-byte.
  - **Golden test** asserting the post-init `.keni/` tree against an explicit list of expected paths (the literal §5.1 prototype subset) — the same list the documentation pins.

## Capabilities

### New Capabilities

- `project-layout`: The on-disk bootstrap contract for a Keni project — what `.keni/` and `~/.keni/` look like after `keni init` runs, the UX and exit conditions of the `keni init` command itself, the rules around `project_id` generation, the `.gitignore` merge policy, the initial-commit policy, the idempotency contract for re-runs, the partial-state repair behaviour, and the explicit list of files / directories the prototype owns under `.keni/` (vs. those reserved for MVP steps 14 and 15). This capability does **not** describe how individual files are read or written (that is `storage`); it describes the shape of the project after init and the deterministic command flow that produces it. Step 13 (`keni start`) and every later step that operates against `.keni/` reads this spec to know what they can rely on existing.

### Modified Capabilities

- `storage`: Adds the `ConfigStore.writeGlobalConfig(config)` requirement and the matching contract-test scenario (write-then-read round-trip; atomic on-disk write for the file adapter). No existing requirement is changed; the shape of `GlobalConfig` is unchanged; the on-disk path (`~/.keni/config.yaml`) is unchanged; the atomicity guarantee, single-writer stance, and error model are unchanged. The delta is one ADDED requirement, no MODIFIED or REMOVED requirements.

## Impact

- **Affected code**:
  - `packages/cli/src/main.ts` becomes the CLI entry point with subcommand dispatch (`init` for now; `start` later in step 13). New module(s) under `packages/cli/src/init/` implement the init flow: argument parsing, target-directory validation, git initialisation, `.keni/` directory creation, `project.yaml` writing via `ConfigStore`, `state.json` skeleton writing, global directory bootstrap, `.gitignore` merge, initial commit. A small `packages/cli/src/git.ts` (or equivalent) wraps the few `git` invocations needed (`git init`, `git status --porcelain`, `git add`, `git commit -m`) so they stay easy to unit-test via `Deno.Command` mocking or a thin abstraction.
  - `packages/shared/src/storage/config/interface.ts` gains `writeGlobalConfig` on the `ConfigStore` interface plus accompanying JSDoc.
  - `packages/shared/src/storage/config/file.ts` and `…/memory.ts` implement `writeGlobalConfig`. The file adapter uses `writeFileAtomic` against `<home>/.keni/config.yaml` (lazy-creates the parent directory via `ensureDir`).
  - `packages/shared/src/storage/config/contract_test.ts` adds the round-trip scenario and a malformed-input rejection scenario.
  - `packages/shared/src/storage/config/file_test.ts` and `…/memory_test.ts` get any adapter-specific coverage (file adapter: temp-file in same dir as target, idempotent writes; memory adapter: deep-clone semantics).
  - `packages/shared/src/storage/README.md` gets a one-paragraph note on `writeGlobalConfig` to keep the contract docs current.
  - `packages/shared/src/storage/mod.ts` requires no new export (the new method is on an already-exported interface; `GlobalConfig` is already exported).
- **Affected APIs / contracts**:
  - **Public**: `keni init` CLI command (new). Inputs: optional positional path, no flags in the prototype. Outputs: exit code (0 on success, non-zero on user-facing error), structured stdout messages, optional `--json` machine-readable mode (deferred to a later step unless trivial to add now).
  - **Internal**: `ConfigStore.writeGlobalConfig(config: GlobalConfig): Promise<void>` (new). Atomicity, error model, single-writer constraint match the existing `writeProjectConfig`.
  - **No change** to: `TicketStore`, `PRStore`, `ActivityLogStore` interfaces; the file-system layout under `.keni/tickets/`, `.keni/prs/`, `.keni/activity/`; the YAML / JSONL on-disk shapes; the four storage error classes; the id-generation module.
- **Affected dependencies**:
  - No new runtime dependency is required — Deno's stdlib already provides everything: `@std/path` (already imported), `@std/fs` for `ensureDir` (already imported), `@std/uuid` for v4 generation (already imported, currently used for v7 in activity ids — adding `v4` is a sub-import). YAML serialisation of the global config stub goes through `@std/yaml` (already imported).
  - **Possibly** `@std/cli` for argument parsing if the prototype's `init` flag-set grows beyond a single positional argument. Otherwise, a hand-rolled switch on `Deno.args[0] === "init"` is sufficient for prototype scope. This decision is recorded in design.md, not pre-judged here.
  - `git` itself is an external system dependency. The README already assumes git is on the path; no version constraint is needed for the operations Keni uses (`init`, `status`, `add`, `commit`).
- **Affected tests**:
  - New: `keni init` integration tests under `packages/cli/` (fresh dir, existing repo, existing non-repo dir, existing `.gitignore`, partial-state repair, idempotent re-run, golden-layout assertion). These tests run inside `Deno.makeTempDir()` and shell out to `git` for the initialisation parts, so they require the test machine to have `git` on the path — already true on CI (Ubuntu) and developer macOS.
  - Extended: `ConfigStore` contract test gains write-then-read round-trips for global config; file adapter test gets a same-directory-tempfile assertion for the global path.
  - Existing tests are untouched. Step 02's 200+ tests remain green.
- **Downstream steps unblocked**:
  - **Step 04 (REST APIs)**, **05 (agents + WS)**, **06 (MCP)**, **07 – 09 (role runtimes, scheduler, engineer)**, **10 – 12 (SPA)**, **13 (`keni start`)**, **14 (PO spec/CR I/O)**, **15 (chat messages)**, **16 – 27** all assume the user has a Keni project on disk to operate against. After this step, every subsequent integration test can run `keni init` in a temp dir and proceed against a real `.keni/` instead of fabricating one with raw filesystem calls.
  - **Step 13 (`keni start`)** specifically gains a sibling subcommand to dispatch to and a guarantee that the project it boots against has a valid `project.yaml` (with a `project_id`) — engineer workspaces under `~/.keni/workspaces/<project-id>/` need that id.
- **Non-impact (deliberate)**:
  - **No `keni start`** — that's step 13. This change ships only the `init` subcommand; `start` may be a placeholder stub that prints a "not implemented" message, or the dispatcher may simply error on unknown subcommands (decision recorded in design.md).
  - **No engineer workspace clones.** `~/.keni/workspaces/` is created lazily by step 09 when the first agent is added. `keni init` does not pre-create it.
  - **No `.keni/de-facto-spec/`, `.keni/changes/`, or `.keni/chat/` directories.** Spec §5.1 marks those as MVP. Steps 14 and 15 add them additively.
  - **No `.env` loading.** Step 27 owns `.env`. This step only ensures `.env` is git-ignored (so a user who already has one is protected from accidental commits).
  - **No prompt embedding work.** Per spec §11#3 prompts ship as TS module exports inside the package that uses them; the `developer-setup` capability already locks that convention at the repo level. No change to prompts ships in this step.
  - **No state.json semantics.** `keni init` writes a skeleton (`{ "watermarks": {} }` or similar minimal shape); the cron scheduler (step 08) and PO chat queue (step 14+) define the real schema.
  - **No remote git, no GitHub integration, no commit signing.** Local git only, matching `spec.md` §5.5 (remote git is post-MVP).
  - **No interactive prompts.** `keni init` is non-interactive in the prototype; if the user has unexpected state (e.g., `project.yaml` exists with a different `project_id` than expected), it errors out and asks the user to resolve manually rather than asking yes/no questions.

## Spec references

- §5.1 — Project folder layout: the exact set of files and directories `.keni/` should contain after init, including `project.yaml`, `state.json` (git-ignored), and the per-artifact directories.
- §5.2 — Global directory layout: `~/.keni/config.yaml`, `~/.keni/workspaces/`, `~/.keni/logs/`. The `<project-id>` rationale ("renaming or moving the project folder does not orphan workspaces") is the reason `project_id` is generated and persisted in `project.yaml` during init.
- §5.3 — `.keni/` write boundary: every write to `.keni/` goes through the API. `keni init` is a one-time scaffolding action that runs *before* the orchestration server is up, so it bypasses MCP/REST and writes directly via `ConfigStore.writeProjectConfig` (which is the storage interface — still going through the abstraction, just not over the wire).
- §7.1 — `keni init` UX: the contract that running `keni init` in any folder produces a working project, including the global-directory bootstrap on first use.
- §2#6 / §11#5 — Files-first storage commitment: this step puts the actual files where the spec said they go, using the storage interfaces that step 02 introduced.
- §11#3 — Prompts-as-code: prompts ship with the binary, never as files. This step does not introduce any prompt files; the convention is preserved.
