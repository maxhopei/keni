## Why

Steps 04, 05, and 06 stood up the orchestration server (REST + WebSocket + agents API) and the engineer MCP surface. What is still missing — and what every later step structurally blocks on — is the **execution loop itself**: the deterministic wrapper that *runs* a coding-agent subprocess inside Keni's `spec.md` §6 lifecycle. Step 08 (cron scheduler) needs *something* to invoke on every tick; step 09 (engineer specialisation) needs a host for the engineer prompt and precheck; step 17 (PO mode selection) needs a place to plug its four-mode arbiter without re-implementing subprocess plumbing; step 19 (PO chat proxy) needs a wrapper that already accepts a `--resume <id>` parameter; step 13 (`keni start`) needs an end-to-end smoke that exercises an actual cycle. Without this step every one of those is structurally blocked: there is no `RoleRuntime` interface to schedule, no shared subprocess-lifecycle code, no shared activity-log emission contract, no shared summary-line capture. `spec.md` §6.2 explicitly names the seven-step cycle (log start → resolve prompt → inject env → spawn → stream → capture summary → conditional `idle` event); this change is the literal implementation of that section, generalised so engineer, QA, and PO can plug into it.

The other reason now is **agent-runner agnosticism** (§6.4). The whole architectural reason for centralising the cycle behind a single `RoleRuntime` is that "Claude Code today, Cursor agent tomorrow, OpenCode the day after" is a one-line config change rather than a rewrite of every role. A second motivation is **prompts as code** (§11#3): bundling the role prompt at compile time — never loaded from `.keni/` — is enforceable only if the runtime owns prompt resolution. Both invariants live in this step or nowhere; if a downstream role lands its own subprocess wrapper, the seam evaporates and §6.4 plus §11#3 become pinky-promises rather than enforced contracts.

Finally, this step is where **the `spec.md` §2 deterministic-wrapper invariants get pinned in code**: §2#2 (fresh session per run — the cycle constructs the subprocess from scratch every time, no shared state across cycles), §2#4 (thin wrapper, agentic decisions — the runtime never decides what the agent works on or interprets output beyond the summary line), and §2#5 (one concern per session — the runtime returns after exactly one cycle). Centralising those in one place means the engineer / QA / PO runtimes downstream cannot accidentally violate them.

## What Changes

- Stand up a new **`role-runtime-common` package surface** under `packages/role-runtimes/src/common/`, exposing a typed `RoleRuntime` interface whose only public method is `startCycle(params: RoleCycleParams): Promise<RoleCycleResult>`. The interface lives in `@keni/role-runtimes` (the existing workspace member, currently a stub); this change populates it with the lifecycle contract every role's runtime extends. The package's `main.ts` re-exports the interface, the cycle implementation, the supporting types, and the bundled-prompt resolution helpers so step 09 (engineer specialisation) and step 17 (PO runtime) can `import { startCycle, type RoleCycleParams, type CodingAgentInvoker } from "@keni/role-runtimes"` without reaching into subdirectories.
- Define the **deterministic cycle algorithm** as a pure-function-shaped pipeline whose seven steps map 1:1 onto `spec.md` §6.2. Pseudocode of the contract:
  1. **Precheck.** Call `params.precheck(prepCtx)` (a role-supplied function) before any HTTP call. If precheck returns `{ kind: "skip", reason }`, return `{ outcome: "precheck_skipped", reason }` immediately without spawning a subprocess and without emitting activity entries — the deterministic-precheck guarantee from §6.1 applies here. If precheck returns `{ kind: "proceed", roleContext }`, proceed.
  2. **Log session start.** `POST /activity` via the activity-log adapter (step below) with `event: "session_start"`, `session_id` (uuidv7 generated here), `summary: <precheck's roleContext.summary or null>`. The orchestration server's `applyActivityEvent` flips the agent's runtime status to `running` and emits `agent.state_changed` automatically.
  3. **Resolve the role's bundled system prompt.** Call `params.promptResolver(prepCtx)` — a role-supplied function returning a `BundledPrompt` (a `{ name: string, body: string }` record whose `body` is a plain string compiled into the binary). The runtime asserts the `body` is non-empty and matches `params.expectedPromptName` when supplied (defence-in-depth against the wrong prompt being plugged in).
  4. **Inject environment context.** Build the `CodingAgentInvocation` value: project name (from `ResolvedConfig`), agent id (from `params.agentId`), workspace path (from `params.workspacePath`, optional — engineer-only today; PO modes pass `null`), MCP server config (from `params.mcpServerConfig`, an explicit `{ command, args, env }` block matching the coding-agent CLI's `mcpServers` shape), env-var allowlist (from `params.envAllowlist`, default `[]`), and the resolved prompt body. The runtime never reads `Deno.env` directly; the allowlist is the only seam.
  5. **Spawn the coding-agent subprocess.** Delegate to `params.codingAgentInvoker.invoke(invocation, lifecycle)` (the role-runtime *itself* hands the invoker a `RoleRuntimeLifecycle` callback set so the invoker reports stdout/stderr lines, exit code, etc., back without knowing about activity-log emission). The default invoker shipped in this step is `createSubprocessCodingAgentInvoker(opts)` which drives `Deno.Command`, but the seam lets a role inject a fake (or, post-MVP, an HTTP-transport invoker).
  6. **Stream stdout / stderr to the activity log.** As the invoker emits `stdout_line` / `stderr_line` callbacks, the runtime calls `POST /activity` with `event: "subprocess_stdout"` / `event: "subprocess_stderr"`, `session_id` matching step 2, `summary: <line truncated to 4 KB minus envelope overhead>`. The orchestration server's existing 4 KB activity-entry cap (§5.1 / step 04 storage layer) is the hard ceiling — lines that exceed it are split client-side. Emission is per-line (not per chunk and not per periodic flush) so the activity log is human-readable when streamed back to the SPA.
  7. **Capture the summary line and log session end.** When the subprocess exits, the runtime extracts the *last non-empty line of stdout* as the session's one-line summary (§6.3). It then calls `POST /activity` with `event: "session_end"`, `session_id` matching step 2, `summary: <captured>`, `refs: { exit_code: <number> }`. The orchestration server flips the agent's runtime status back to `idle` and emits `agent.state_changed`.
  - **Conditional `idle` emission.** If the subprocess exited within `IDLE_THRESHOLD_MS` (default 250 ms) *and* produced no non-empty stdout line, the runtime suppresses `session_end` and instead appends `event: "idle"` (§6.2#7). The activity log thus distinguishes "agent ran and finished" from "agent had nothing to do" without burning UI noise on quiet ticks.
- Ship a **subprocess-utility module** (`packages/role-runtimes/src/common/subprocess.ts`) that wraps `Deno.Command` with the §6 invariants:
  - **Graceful termination.** `terminate(child, { graceMs }) → Promise<{ exitCode, terminatedBy: "exit" | "sigterm" | "sigkill" }>`. Sends SIGTERM, waits up to `graceMs` (default 5 000), then sends SIGKILL if the process is still alive. On Windows where SIGTERM is unsupported, `Deno.Command.kill()` is a hard kill; the utility documents the platform difference and emits a one-line warning the first time it is called on Windows. Default graceMs is 5 000; configurable per role via `RoleCycleParams`.
  - **Exit-code mapping.** A normal exit returns `{ outcome: "completed", exitCode }`; a non-zero exit returns the same shape with the non-zero code so the role's caller can decide whether to log it as an error in the activity entry's `refs.exit_code`.
  - **Env-var propagation.** Only env vars in the role's `envAllowlist` are forwarded to the child; everything else is stripped (no `Deno.env.toObject()` leak). The MCP-config env (`KENI_MCP_AGENT`, `KENI_MCP_SERVER_URL`, `KENI_MCP_WORKSPACE`) is added in addition.
- Ship a **bundled-prompt resolution mechanism** that satisfies §11#3 ("prompts as code"). Prompts live as TypeScript string constants under `packages/role-runtimes/src/common/prompts/` (one file per prompt — e.g., `engineer.ts`, `po-conversation-to-cr.ts` — each `export const PROMPT_BODY = ` followed by a tagged-template string). A role's `promptResolver` simply imports the constant and wraps it in a `BundledPrompt`. This step ships *one* placeholder prompt file (`placeholder.ts`) used by the integration test; step 09 lands the engineer prompt and step 18 lands the four PO prompts. The mechanism rejects loading from any path under `.keni/` or `~/.keni/` — there is no file-IO seam; the only way a prompt arrives at the runtime is by being a TS string constant compiled in.
- Ship an **activity-log adapter** (`packages/role-runtimes/src/common/activityClient.ts`): a minimal typed HTTP client whose only methods are `appendSessionStart(...)`, `appendSessionEnd(...)`, `appendIdle(...)`, `appendSubprocessOutput(streamKind, line, ...)`, and a low-level `appendRaw(input)` for tests. Each method calls `POST /activity` carrying `X-Keni-Role: <role>` and `X-Keni-Agent: <agentId>` (stamped from the cycle params). The adapter does *not* know about MCP tools (engineers go through the MCP server's `append_activity_entry` already; this is for the runtime's own emission) and does *not* duplicate any wire shape — it imports `ActivityAppendRequest` from `@keni/shared/wire/`.
- Ship a **`CodingAgentInvoker` interface** plus a default `createSubprocessCodingAgentInvoker(opts)` factory. The interface is `invoke(invocation: CodingAgentInvocation, lifecycle: CodingAgentLifecycle): Promise<CodingAgentOutcome>`; the lifecycle bag exposes `onStdoutLine`, `onStderrLine`, `onSpawn(child)`, and an `abortSignal`. The factory produces an invoker that reads the resolved CLI binary from `params.cliBinary` (e.g., `"claude"`), shells out via `Deno.Command`, attaches stdin (the prompt), and uses the subprocess utility's termination / streaming primitives. The factory is the seam step 09 plugs to add engineer-specific argument shapes (e.g., `--mcp-config <tempfile>` for `claude`); the runtime itself is binary-agnostic.
- Define **`resume_session_id` forwarding** as part of the cycle contract: `RoleCycleParams.resumeSessionId?: string` is plumbed into `CodingAgentInvocation.resumeSessionId` and forwarded to the invoker. The default subprocess invoker passes it as `--resume <id>` (the convention `claude` and `cursor-agent` use); a role can override the flag shape via `cliResumeFlag: string` (default `--resume`). Step 19 (PO chat-mode CLI proxy) consumes this end-to-end without needing to refactor the cycle. This step accepts the parameter, plumbs it through, and exercises it in a unit test using a fake invoker; no real `--resume` semantics are tested until step 19.
- Define the **cycle's outcome shape**: `RoleCycleResult = { outcome: "completed", sessionId, exitCode, summary } | { outcome: "idle", sessionId } | { outcome: "precheck_skipped", reason } | { outcome: "terminated", sessionId, terminatedBy: "sigterm" | "sigkill", exitCode } | { outcome: "spawn_failed", error }`. The five outcomes are exhaustive; downstream callers (step 08's scheduler, step 17's mode selector) `switch` on `outcome` with type-narrowing.
- Wire **unit and integration tests**:
  - **Unit, with a fake subprocess invoker.** The fake records the `CodingAgentInvocation` it received, lets the test push `stdout` / `stderr` lines, and lets the test specify the exit code (or a thrown spawn failure). Every cycle outcome is exercised: completion, idle, precheck_skipped, terminated (the test calls into the lifecycle's abort signal), spawn_failed.
  - **Unit, prompt resolution.** The placeholder prompt is loaded; expected name mismatch throws; an empty body throws.
  - **Unit, activity-log adapter.** Each method composes the right URL and headers; non-2xx responses surface as a typed `RoleRuntimeHttpError`.
  - **Unit, subprocess utility.** Graceful termination on a slow child; SIGKILL after grace period; exit-code passthrough; env-allowlist filtering.
  - **Integration test.** `Deno.makeTempDir()` project + a real orchestration server on `port: 0`. The "coding agent" is a tiny Deno script that prints a configurable number of stdout lines (parameterised via env vars), then prints the summary line, then exits 0 (or a non-zero code, when the test wants it). The test exercises: a happy-path cycle (asserting `.keni/activity/<date>.jsonl` grew by exactly the expected number of `session_start`, `subprocess_stdout`, `session_end` entries with the right `agent` / `role` / `summary` values), an idle cycle (subprocess exits in 0 ms with no stdout — assert `idle` event is the only one logged), and a graceful-termination cycle (the cycle calls `abort()` mid-run; assert `session_end` with the captured terminate reason is logged). Step 09 will write a real engineer integration test on top of this; step 13's end-to-end smoke is unaffected.
- Define a **`role-runtime` capability spec** that documents the lifecycle contract — every step of the cycle, every parameter, every outcome, the prompt-resolution rules, the subprocess-utility behaviour, the activity-log emission rules, and the `resume_session_id` forwarding. The spec is the document step 09 reads when writing the engineer runtime, step 17 reads when adding mode selection, and step 19 reads when wiring chat-mode resume.
- Update the root `README.md` with a brief paragraph naming the role-runtime layer, where it lives, and the fact that step 09 (engineer) is the first concrete consumer.

## Capabilities

### New Capabilities

- `role-runtime`: The shared lifecycle contract for any Keni role's per-cycle runtime. Names the `RoleRuntime` interface, the `startCycle(params)` cycle algorithm step-by-step (per `spec.md` §6.2), the parameter shapes (`RoleCycleParams`, `CodingAgentInvocation`, `BundledPrompt`, `CodingAgentInvoker`, `CodingAgentLifecycle`), the outcome shapes (`RoleCycleResult` discriminated union), the bundled-prompt resolution rule (TS string constants only — no path under `.keni/` or `~/.keni/`), the subprocess utility's graceful-termination contract (SIGTERM grace → SIGKILL, configurable grace period), the activity-log emission contract (per-line streaming, hard 4 KB cap, `session_start` / `subprocess_stdout` / `subprocess_stderr` / `session_end` / `idle` events), the `resume_session_id` forwarding contract, the env-allowlist contract, the explicit non-goals (no scheduler — step 08; no engineer-specific concerns — step 09; no PO mode selection — step 17), and the `RoleRuntimeHttpError` shape for activity-log failures.

### Modified Capabilities

None. The `orchestration-server` capability is consumed without modification — every cycle's activity-log emission goes through `POST /activity` carrying `X-Keni-Role` and `X-Keni-Agent`, and the agent-state transitions piggyback on the existing `applyActivityEvent` rules from step 05. The `mcp-engineer-surface` capability is unchanged — the runtime never speaks MCP itself; it only spawns coding-agent subprocesses that *do*. The `storage` and `project-layout` capabilities are unchanged — the runtime never reads or writes `.keni/` directly. The `ErrorCode` enum is unchanged.

- **Downstream steps unblocked**:
  - **Step 08 (cron scheduler)** gets `startCycle` to invoke per tick. The scheduler reads `paused` from the agents API, runs the role's precheck (now passed *into* `startCycle`), and on a `precheck_skipped` outcome budgets zero LLM tokens. The cycle's emit-`agent.state_changed`-via-activity-log seam means the scheduler does not need to emit those events itself.
  - **Step 09 (engineer specialisation)** gets a host for the engineer prompt and precheck plus a workspace-aware integration story. The engineer's `RoleCycleParams` plumb the `workspacePath` to `CodingAgentInvocation` and the engineer prompt body lives in `packages/role-runtimes/src/engineer/prompt.ts` (new in step 09) imported as a TS constant.
  - **Step 13 (`keni start`)** ships unchanged. The runtime is invoked by the scheduler (step 08), not by `keni start` directly.
  - **Step 17 (PO mode selection)** layers a four-mode arbiter *into* the precheck function rather than refactoring the cycle: PO precheck inspects the conversation-to-CR queue / CR statuses / ticket statuses on every tick and returns either `{ kind: "skip", reason: "no_mode_applicable" }` or `{ kind: "proceed", roleContext: { mode: "verify_and_fold", ... } }`; the prompt resolver then picks the bundled prompt for the chosen mode. Zero changes to `startCycle`'s shape.
  - **Step 19 (PO chat-mode CLI proxy)** consumes `resume_session_id` end-to-end. The chat handler computes the session id from `state.json`, hands it as `RoleCycleParams.resumeSessionId`, and the cycle forwards it to the subprocess invoker. Step 19's only addition is the chat-specific precheck (event-driven, not scheduled) and the structured-output parse for the new session id; the runtime itself is unchanged.
  - **Step 26 (multi-engineer)** is unaffected. Multiple engineers means multiple concurrent `startCycle` calls in different schedulers; the runtime is stateless across calls (each cycle spawns a fresh subprocess), so concurrency lives at the scheduler layer.
- **Non-impact (deliberate)**:
  - **No scheduler.** This step ships a single function (`startCycle`) that runs *one* cycle. Looping is step 08.
  - **No engineer specialisation.** The placeholder prompt and the no-op precheck are stand-ins; the engineer prompt and precheck land in step 09. The runtime does not import anything engineer-shaped.
  - **No PO mode selection.** Step 17 plugs mode selection into the precheck; the cycle has no awareness of "modes".
  - **No coding-agent CLI specifics.** The `cliBinary` is a string passed in `RoleCycleParams`; the resume flag, mcp-config flag, and prompt-injection mechanism are configurable via the invoker's `opts` parameter. This step ships a generic `Deno.Command`-based invoker with documented defaults; CLI-specific adapters are downstream.
  - **No `--mcp-config` file management.** The cycle hands the invoker a `mcpServerConfig` object; the invoker decides how to surface it to the CLI (write to a temp file, use an env var, etc.). The default invoker writes a JSON file under `Deno.makeTempDir()` and passes `--mcp-config <path>`; the file is cleaned up on cycle exit (success or failure) via `try`/`finally`.
  - **No live SPA push.** Activity-log entries flow through the orchestration server's existing event bus and `/events` WebSocket; the runtime does not open its own socket.
  - **No persistent runtime state.** A cycle holds no state across invocations (per `spec.md` §11#2 fresh-session rule). The session id is freshly generated per cycle (uuidv7 from `@std/uuid`); there is no continuation across cycles other than the explicit `resume_session_id` parameter for chat mode.
  - **No cancellation infrastructure beyond `AbortSignal`.** `RoleCycleParams.signal: AbortSignal | undefined` lets the scheduler (step 08) abort an in-flight cycle. Aborting causes the runtime to terminate the subprocess via the subprocess utility's `terminate(child, { graceMs })`. The signal is plumbed through; step 08 owns the `setTimeout`-driven timeout that fires it.
  - **No retries.** A `spawn_failed` outcome is returned to the caller; the cycle does not retry. Step 08 may schedule a retry on the next tick, but that's its concern.
  - **No log-level filtering.** Every stdout/stderr line is emitted as an activity entry. If this proves chatty, step 09 may add a per-role filter; the cycle does not opine.
