## Why

Today a user who has run `keni init` cannot reach the prototype's promised loop in one move: they must `deno run -A packages/server/src/main.ts --project <abs-path> --port 0` for the orchestration server, `cd packages/spa && KENI_SERVER_URL=http://127.0.0.1:<port> deno task dev` in a second terminal, and remember to copy the printed port between the two; the SPA bundle is not served by the orchestration server, the layered config (`.keni/project.yaml` plus `~/.keni/config.yaml`) is read piecemeal by `runServer` only (no merge with the global file), `.env` is invisible to every Keni process, port-conflict recovery is absent (a busy `--port 8080` aborts with a raw `Deno.serve` `EADDRINUSE`), no health endpoint exists for the smoke test or for forthcoming process-supervision tooling, and a SIGINT trickles through `runServer`'s shutdown without coordinating the scheduler's interrupt path with the SPA-bundle file handles or any in-flight subprocess in a documented, repeatable way. Step 13 closes that gap: ship `keni start [project-path]` as a single CLI entry point, host the production SPA bundle from the orchestration server itself (with a documented dev-mode fallback that proxies to Vite), pick a port from a configured range with deterministic conflict recovery, expose a tiny `GET /health` for the smoke test (and for future supervisors), wire layered config + minimal `.env` loading at the seam future step 27 will own, and on SIGINT/SIGTERM run a documented graceful-shutdown sequence (pause the scheduler → interrupt every in-flight cycle → flush the activity log → close the listener). This step is the prototype's exit criterion (`spec.md` §8): after it lands, a user can `keni init` an empty folder, `keni start`, open the printed URL, create a ticket, and watch the engineer drive it through `in_progress → … → ready_for_test` without writing any other invocation by hand.

## What Changes

- **NEW CLI subcommand `keni start [project-path]`** on `@keni/cli`. Resolves the project path (default: `cwd`); fails fast (exit 1) if the folder lacks `.keni/project.yaml`; loads the layered config (project then global, project wins on key collision); loads `.env` from the project root via a minimal in-house loader (documented seam for the heavier `.env` UX in step 27); resolves a TCP port from the configured range (default `7777..7787`) with `EADDRINUSE` fallback to the next port in the range; resolves the SPA-serving mode (`bundled` for production, `dev-proxy` when the project's `~/.keni/workspaces/...` SPA dev server is targeted via `--spa-dev-url`, default `bundled`); calls into `runServer` via the existing in-process composition (no second process); prints `Keni server running at http://127.0.0.1:<port>` in the documented format; honours pause state from `.keni/project.yaml`'s `paused_agents` (when present, additive to the existing `agents` field) on boot; installs SIGINT and SIGTERM handlers that drive the documented graceful-shutdown sequence; returns exit code 0 on clean shutdown, 1 on filesystem / git / boot failure, 2 on usage error.
- **Static SPA serving** mounted on the orchestration server when `ServerDeps.staticAssetsRoot` is supplied. The new route group serves `/` (the SPA bundle's `index.html`), `/assets/*` (immutable hashed files with a one-year `Cache-Control: public, max-age=31536000, immutable` header), and any unmatched non-`/api`, non-`/events` GET path SHALL fall through to `index.html` so the SPA's client-side router (`react-router-dom` `BrowserRouter`) handles deep links. The route group is **inactive by default** — `runServer` (and every test that builds `createServer(deps, opts)`) keeps the existing behaviour when `staticAssetsRoot` is absent.
- **NEW health endpoint `GET /health`** on the orchestration server. Returns `200 { data: { status: "ok", project_id, uptime_ms, version }, project_id }` to any caller (no role-header requirement — the only response that bypasses the role guard, with the rationale documented inline). Unhealthy boots (e.g., a corrupt `project.yaml`) never reach this endpoint because the server hasn't started; once the server is up, `/health` is unconditionally `200`. The endpoint is the smoke-test seam and the future supervisor seam.
- **Layered config loader** added to `@keni/cli`: a `loadKeniConfig({ projectDir, homeDir, env })` helper that merges `~/.keni/config.yaml` (when present) with `.keni/project.yaml`. Project wins per key. The loader returns the merged `ProjectConfig` plus a separate `KeniStartConfig` (port range, SPA mode, future fields) so existing callers of `ConfigStore.readProjectConfig()` are unchanged. The loader is a pure function — `keni init` does NOT change.
- **Minimal `.env` loader** added to `@keni/cli`: reads `<projectDir>/.env` (when present), parses `KEY=VALUE` (one per line, comments allowed via `#`, double-quoted values supported), and overlays the parsed entries into the orchestration-server subprocess's `Deno.env` cache **inside the same `runServer` invocation** (no actual subprocess — `keni start` calls `runServer` in-process). The loader exposes the parsed map for tests; it does NOT support multiline values, command substitution, or variable interpolation (those land in step 27).
- **Graceful-shutdown sequence** documented and implemented. On SIGINT / SIGTERM, `keni start`: (1) signals the scheduler to stop accepting new ticks (idempotent `scheduler.stop()` already specced); (2) iterates every roster entry whose runtime status is `running` and calls `scheduler.interrupt(agentId)` (per the existing `interrupt-and-timeout-ux` capability — SIGTERM → grace → SIGKILL, the activity post lands a `session_interrupted`); (3) awaits a configurable shutdown grace period (default `2_000` ms; `10_000` ms hard cap) before continuing; (4) calls the existing `await serverHandle.abort()` to stop accepting connections; (5) returns exit code `0`. A second SIGINT during shutdown short-circuits to exit code `130` (the conventional SIGINT exit code for forced shutdown).
- **Documentation**: README's existing "Run the orchestration server" subsection gains a new **Quickstart with `keni start`** subsection naming the new subcommand as the first-class user entry point and demoting the direct `deno run` invocation to a "Direct invocation (development)" sub-subsection. README also gains a new top-level **End-to-end smoke test** subsection — the four-step runbook (`init`, `start`, open URL, drive a ticket) that is the prototype's documented exit criterion — and a one-line cross-link from the existing "Getting started" section pointing to it.
- **Out of scope** (deferred to other steps): rich `.env` UX (multiline values, variable interpolation, `.env.local` precedence) — step 27. Multi-project orchestration in one server — out of MVP. Authentication / TLS — local-only by design. Process supervision (auto-restart, log rotation beyond what `runServer` already does) — post-MVP. Bundling the `keni` binary — post-MVP (the prototype invokes the CLI via `deno run -A packages/cli/src/main.ts start`).

## Capabilities

### New Capabilities

- `cli-start`: The boot contract for `keni start [project-path]` — argv parsing, layered config + `.env` loading, port selection with conflict recovery, SPA-serving mode resolution, in-process composition with `runServer`, the documented graceful-shutdown sequence, and the exit-code table. This capability is the seam every later operational change (process supervision, log routing, multi-project switcher) extends.

### Modified Capabilities

- `orchestration-server`: Adds the optional static-SPA route group (mounted only when `ServerDeps.staticAssetsRoot` is present), the SPA-route fallthrough rule for `react-router-dom` deep links, and the unconditionally-`200` `GET /health` endpoint (with its single, documented exemption from the role-identity middleware).
- `developer-setup`: README's "Getting started" / "Run the orchestration server" / "Run the SPA" subsections are restructured: a new "Quickstart with `keni start`" subsection becomes the first-class user entry point; the existing direct-`deno run` invocations are demoted to "Direct invocation (development)" sub-subsections; a new top-level "End-to-end smoke test" subsection captures the prototype's exit-criterion runbook.

## Impact

- Code:
  - `packages/cli/src/start/` (new) — `mod.ts` (the `runStart` entry point invoked by the dispatcher), `args.ts` (argv parsing for `start`), `loadConfig.ts` (the layered-config helper), `loadEnv.ts` (the minimal `.env` parser), `port.ts` (the port-range + EADDRINUSE retry), `shutdown.ts` (the SIGINT/SIGTERM sequence and the in-flight-interrupt loop), `messages.ts` (additions for `keni start`'s error / runbook copy), plus matching `*_test.ts` files for each.
  - `packages/cli/src/main.ts` — extend the dispatcher's `switch (subcommand)` with a `case "start": return await runStart(parseStartArgs(rest), io);` arm, mirroring the existing `case "init":` shape.
  - `packages/server/src/createServer.ts` and `packages/server/src/runServer.ts` — extend `ServerDeps` with `staticAssetsRoot?: string` and `serverStartedAt?: Date` (consumed by `/health` for `uptime_ms`); the static-asset router mounts only when `staticAssetsRoot` is provided.
  - `packages/server/src/routes/health.ts` (new) and registration in the existing routes barrel — the `/health` handler reads from `agentRuntimeStateStore` + the new `serverStartedAt` to populate the response body. No runtime-state mutation.
  - `packages/server/src/routes/static.ts` (new) — the static-asset middleware + the SPA-fallthrough handler, mounted via `app.get("/", ...)`, `app.get("/assets/*", ...)`, and a catch-all middleware that serves `index.html` only for non-`/api`-prefixed, non-`/events`, non-`/agents`, non-`/tickets`, non-`/prs`, non-`/activity`, non-`/health` GET paths (the discriminator list is encoded as a closed allowlist of the existing route prefixes so adding a new REST prefix later forces a code change).
  - `packages/cli/deno.json` — add a `start` task that runs `deno run -A src/main.ts start` against the workspace root for developer convenience (mirrors `init`).
  - `README.md` — restructure the existing "Getting started" / "Run the orchestration server" / "Run the SPA" subsections per the proposal; add the new "End-to-end smoke test" subsection as a top-level peer to "Conventions".
- Wire shapes: One new envelope `HealthResponse = { data: { status: "ok", project_id, uptime_ms, version }, project_id }` exported from `@keni/shared/wire/`. No new `EventName`, no new `ErrorCode`. The existing `EventFrame` discriminated union is unchanged.
- Dependencies: No new npm / jsr packages. The `.env` parser is in-house (≤50 lines).
- Documentation: The two README subsections plus the new "End-to-end smoke test" runbook section. The capability spec for `cli-start` is the canonical reference for every behavioural detail.
- Tests: Unit tests for each new module in `packages/cli/src/start/` (config merge, env parse, port-range fallback, args parsing, shutdown sequencing). New orchestration-server route tests for `/health` (4 cases) and `/static` (5 cases — index, asset, fallthrough for SPA route, no-fallthrough for `/api`-prefix, missing `staticAssetsRoot` → 404 on `/`). One end-to-end smoke test in `packages/cli/src/start/start_e2e_test.ts` that boots `keni start` against a fixture project, asserts the printed URL responds `200` on `/health`, asserts the SPA bundle's `index.html` is served on `/`, and tears down via the injected `AbortSignal`.
