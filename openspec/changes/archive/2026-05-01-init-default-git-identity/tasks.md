- [x] 1.2 Add a private async helper `readGitConfigValue(cwd: string, key: string): Promise<string | null>` that runs `git config <key>` against `cwd`, returns `null` when git exits 1 with empty stderr (the documented "key is unset" path), returns the trimmed stdout value when git exits 0 and stdout is non-empty, returns `null` when git exits 0 with whitespace-only stdout, and throws `GitOperationError` on any other failure mode (any non-zero code with non-empty stderr, or any thrown subprocess error).
- [x] 1.3 Add a private async helper `resolveCommitIdentityEnv(cwd: string): Promise<Record<string, string> | undefined>` that calls `readGitConfigValue(cwd, "user.name")` and `readGitConfigValue(cwd, "user.email")`, returns `undefined` when both are non-null and non-empty (the "honour user identity" path), and otherwise returns the four-key map `{ GIT_AUTHOR_NAME: "Keni", GIT_AUTHOR_EMAIL: "keni@example.invalid", GIT_COMMITTER_NAME: "Keni", GIT_COMMITTER_EMAIL: "keni@example.invalid" }` (the "fallback" path).

No other change to `defaultCommit`'s shape.
- [x] 1.5 Update the module-level doc comment at the top of `packages/cli/src/init/git.ts`. Replace the existing line "No global config is touched; the user's `git config user.name` / `user.email` are honoured" with a paragraph that names the additive fallback contract: identity is honoured when configured (any layer); when neither `user.name` nor `user.email` is reachable, a per-invocation fallback identity (`Keni <keni@example.invalid>`) is supplied via `GIT_AUTHOR_*` / `GIT_COMMITTER_*` env vars on that single subprocess, and no persistent config (global, local, system, or XDG) is written. ## Spec walk verification

| Scenario | Pin |
| --- | --- |
| **Configured user identity is honoured** | `init_integration_test.ts` :: `fresh empty dir produces full layout, single commit, valid UUIDv4 project_id` (uses `withEnv` only — picks up host `~/.gitconfig`, must succeed under the user's real identity); `git_test.ts` :: `commit honours configured user identity even when GIT_CONFIG_GLOBAL is /dev/null` (per-repo identity wins over a `/dev/null` global, isolating the "honour" path from the host's `~/.gitconfig`). |
| **Missing user identity falls back to the documented Keni identity** | `git_test.ts` :: `commit produces a Keni-fallback identity when no user.name / user.email is configured` (unit-level); `init_integration_test.ts` :: `produces a Keni-fallback initial commit when no git identity is configured` (end-to-end through `runInit`); plus task 4.2's manual smoke test (`HOME=$(mktemp -d) GIT_CONFIG_GLOBAL=/dev/null GIT_CONFIG_SYSTEM=/dev/null deno run -A packages/cli/src/main.ts init`). |
| **Fallback does not persist any git config** | `git_test.ts` :: `commit fallback does not write any persistent git config` (asserts `git config --get user.name`/`user.email` exit 1 with empty stdout, and that `.git/config` contains no `[user]` section). Reinforced at end-to-end level by task 4.2's manual `grep -c '\[user\]' <tempDir>/.git/config` returning 0. The "host's `~/.gitconfig` is unchanged" sub-clause is structural: `defaultCommit` only ever calls `runGit("commit", ["-m", message], cwd, identityEnv)` and `runGit("config", [key], cwd)` (read-only) — no `git config --set` / no file write touches `~/.gitconfig`. |
| **Partially-configured identity falls back rather than producing an inconsistent commit** | Structural pin via `resolveCommitIdentityEnv` in `git.ts`: the predicate is `if (name !== null && email !== null) return undefined;` — `undefined` (honour-user) requires *both* keys non-null. Any partial state (one-of-two unset, or one-of-two whitespace-only) flows through `readGitConfigValue` returning `null` for the missing/whitespace key, so the four-key fallback map is returned. The "no persistent config written" sub-clause inherits from the previous scenario's pin. An explicit unit test for partial state was deemed over-coverage given the predicate's shape, but the test slot is open for a follow-up if the predicate is ever refactored. |
| **Subsequent commits in the project use whatever identity git resolves** | Structural pin via `defaultCommit` in `git.ts`: the env-var override is supplied as the fourth argument to `runGit` on a single subprocess invocation; no `git config --set`, no file write touches `.git/config`. Reinforced by `git_test.ts` :: `commit fallback does not write any persistent git config` — once that test passes, any subsequent `git commit` in the same repo runs through git's normal resolution chain, which now (assuming the user has since added `user.name`/`user.email`) returns the user's identity. The wider behaviour ("the user adds identity AFTER `keni init` and the next commit is attributed to them") is git's own behaviour, not Keni-specific, and is therefore not re-tested here. |
| **Idempotent re-run with the fallback path is still a no-op** | `init_integration_test.ts` :: `idempotent re-run on a clean project produces no new commits` (existing test, runs under `withEnv`). The new fallback-path test in 3.2 produces a single commit and asserts `git status --porcelain` is empty after; combined with the existing idempotency test's "no new commits on re-run" pin, the same code path (`runInit` → if-already-initialised short-circuit) handles both identity-configured and identity-less hosts transparently — `runInit`'s decision tree branches on filesystem state, not on identity. Migrating the existing idempotency test to also run under `withGitIdentityIsolated` MAY be done in a future change but adds no new coverage today. |

## Hand-off

- **Step 09 (engineer workspaces).** When step 09 ships, engineer-driven commits in workspace clones SHALL NOT inherit this fallback. Step 09 designs its own author policy (likely the agent's id, recorded in the project config, propagated through whatever git client the role runtime uses). The fallback in `git.ts` is `keni init`-local — it lives inside `defaultCommit` and is reachable only via the `GitClient` returned by `createDefaultGitClient()` from this module. Step 09 SHOULD construct its own `GitClient` (or use a different commit primitive entirely) for engineer-workspace commits.
- **Step 25 (manual override).** User-driven commits via the manual-override flow SHALL NOT inherit this fallback. Step 25's commits are attributed to the user (whose identity is by definition configured at that point — the user has typed at the keyboard, and a non-configured-identity user has been told via README to configure git first). If step 25 needs to invoke `git commit` programmatically, it SHALL use a git surface other than `defaultCommit` from this module, or accept that the fallback would fire in the corner case where the user nuked their `~/.gitconfig` between `keni init` and the override action.
- **Any future Keni surface that runs `git commit`.** SHALL NOT import `resolveCommitIdentityEnv` from `git.ts` (the helper is file-private — not exported, not in the `GitClient` interface, deliberately not part of the public type surface). SHALL design its own identity policy if it has one, or rely on the user's configured git identity (with a clear failure mode) otherwise. If a future surface decides it wants the same fallback, the right move is to lift `resolveCommitIdentityEnv` into a shared module and adjust this change's spec to reference it from the new owner — not to copy-paste the constants.
- **The `FALLBACK_IDENTITY` constant.** Lives at the top of `packages/cli/src/init/git.ts` as a `const … as const` at module scope. Pinned to exactly `Keni` and `keni@example.invalid`. Two fallback-path tests assert against these exact strings (drift-checked in task 7.4). The display name `Keni` matches the project name; the email's `.invalid` TLD is reserved by RFC 2606 and is unambiguously non-routable. Changing either value SHALL ship as its own OpenSpec change (the `project-layout` capability spec normatively names both strings).
