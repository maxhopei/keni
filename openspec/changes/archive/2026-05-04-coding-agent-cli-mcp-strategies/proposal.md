## Why

The closed `codingAgentCliRegistry` shipped by the in-flight `engineer-runner-production-wiring` change models all three supported CLIs (`claude`, `cursor-agent`, `codex`) with a single uniform shape: the runtime writes the `mcpServers` config to a tempfile and the entry's `buildArgs` passes it via `--mcp-config <path>`. This works for `claude` (which documents `--mcp-config <file or string>`) but is wrong for the other two:

- **`cursor-agent`** does not accept a `--mcp-config` flag at all (verified against the installed `cursor-agent v2026.04.15-dccdccd` and the [Cursor CLI docs](https://cursor.com/docs/cli/mcp)); it loads MCP servers from `<workspace>/.cursor/mcp.json` (project) or `~/.cursor/mcp.json` (global) and uses `--approve-mcps` to skip approval prompts in headless mode. A user who configured `coding_agent_cli: cursor-agent` and started an engineer hit `error: unknown option '--mcp-config'` and the cycle exited with `subprocess_stderr` immediately, before any tool call.
- **`codex`** also does not accept a `--mcp-config` flag (per [OpenAI Codex MCP docs](https://developers.openai.com/codex/mcp) and the closed feature request [openai/codex#9550](https://github.com/openai/codex/issues/9550)); it reads MCP servers from `~/.codex/config.toml` / `<project>/.codex/config.toml` (TOML, not JSON) and the documented per-session `-c key=value` overrides have known bugs (e.g. [openai/codex#16045](https://github.com/openai/codex/issues/16045)).

The original change explicitly tagged the `cursor-agent` and `codex` entries `coverage: "best-effort"` and tracked the integration-test gap in tasks 6.1 / 6.2. This change closes that gap for `cursor-agent` (the user's actual configured CLI) and lays the same per-CLI MCP-config-materialisation seam that `codex` will also need, so a future user configuring `codex` doesn't hit the same class of failure.

While we're touching the registry, the user has asked for a structural cleanup: today every CLI entry lives inline in `codingAgentCliRegistry.ts`. With three entries that share little of their argv shape, splitting each entry into its own file keeps the abstract registry types in one place and makes adding a fourth CLI a single-file diff.

## What Changes

- **Extend the `CodingAgentCliEntry` shape with a per-CLI MCP-config materialisation seam.** Add `mcpConfigStrategy: McpConfigStrategy` where `McpConfigStrategy` is a discriminated union covering `{ kind: "tempfile-json" }` (the existing default — write `{ mcpServers: { keni: ... } }` to `Deno.makeTempFile(...)`, pass the path to `buildArgs`, remove on cycle exit), `{ kind: "workspace-json", relativePath: string, mergeKey: string }` (write to `<workspace>/<relativePath>` merged into the file's existing top-level object under `mergeKey.<entryName>`; no cleanup — the per-agent workspace is keni-managed), and `{ kind: "workspace-toml", relativePath: string, tableHeader: string }` (TOML equivalent for `codex`'s `mcp_servers.<name>` table syntax). Each strategy is a value type — no closures — so the registry stays declarative and a structural test can pin the closed set of strategies.
- **Fix the `cursor-agent` registry entry.** Replace `buildArgs: () => ["--print", "--mcp-config", path]` with `buildArgs: (invocation) => ["--print", "--approve-mcps", "--workspace", invocation.workspacePath ?? "."]`, set `mcpConfigStrategy: { kind: "workspace-json", relativePath: ".cursor/mcp.json", mergeKey: "mcpServers" }`. The `keni` entry is merged into any existing `<workspace>/.cursor/mcp.json` so a user who has committed their own MCP servers in their project repo keeps them in their per-agent workspace.
- **Fix the `codex` registry entry.** Replace `buildArgs: () => ["exec", "--mcp-config", path]` with the documented `codex exec` argv shape (no `--mcp-config`), set `mcpConfigStrategy: { kind: "workspace-toml", relativePath: ".codex/config.toml", tableHeader: "mcp_servers" }`. Mark this entry `coverage: "best-effort"` (no `codex` binary on the maintainer's machine; the registry test pins the value-type shape and the modelled argv, the integration-test follow-up under task 6.2 of the prior change remains open).
- **Pin the `claude` entry against the real binary.** The current shape `["--print", "--mcp-config", path]` works against the installed `claude v1.0.2`; document the version-skew note for `claude >= 2.x` (where `--mcp-config` is variadic — see [anthropics/claude-code#16122](https://github.com/anthropics/claude-code/issues/16122)) in the entry JSDoc. Strategy stays `{ kind: "tempfile-json" }`.
- **Set the engineer subprocess `cwd` to the per-agent workspace.** The default subprocess invoker today calls `new Deno.Command(bin, { args, env, stdin, stdout, stderr })` with the parent's `cwd`. Add `cwd: invocation.workspacePath ?? <parent cwd>` so `cursor-agent` (and `codex`, and any future CLI) discovers the project-scoped MCP / config files we just wrote, and so the engineer's filesystem operations land in its workspace by default. This is a behavioural change but matches the engineer-runtime spec's existing `KENI_MCP_WORKSPACE` env-var semantics and the `WorkspaceProvisioner.workspacePathFor(...)` contract.
- **Refactor the registry to one file per CLI.** Move the `claudeEntry` / `cursorAgentEntry` / `codexEntry` constants out of `codingAgentCliRegistry.ts` and into `packages/role-runtimes/src/common/codingAgentClis/{claude,cursorAgent,codex}.ts`. Keep `codingAgentCliRegistry.ts` as the assembly point — it imports each per-CLI module and exports the constant. The `CodingAgentCliEntry` interface, the `McpConfigStrategy` union, the `KnownCli` type, and the `isKnownCli` guard stay in `codingAgentCliRegistry.ts`. Per-CLI files SHALL NOT import each other.
- **Add an integration test for `cursor-agent`** (closes the in-flight `engineer-runner-production-wiring/tasks.md#6.1`). The test boots a fake MCP server on a free port, writes a temp `~/.cursor/` (so the test does not touch the real user's home) via `HOME` overrides, runs the real `cursor-agent` binary against the registry entry, and asserts the MCP server received the expected `tools/list` call. The test SHALL be skipped (with a clear `Deno.test.ignore` reason) when `cursor-agent` is not on `PATH`, so CI without the binary still passes; the maintainer's machine has the binary installed and exercises the full path locally.
- **Update the documented argv shape in the engineer-runtime spec.** Today the spec text says `cursor-agent` and `codex` "SHALL similarly cover ... MCP-config consumption per each CLI's documented contract" without naming the contract. Replace this with the explicit per-CLI argv shape and the explicit MCP-config strategy now that we know the contracts.

This change is **non-breaking** to existing users: a project whose `coding_agent_cli` is unset, or set to `claude`, sees identical behaviour. A project configured for `cursor-agent` goes from "spawns and immediately fails with `unknown option '--mcp-config'`" to "spawns, loads MCP servers, runs". A project configured for `codex` goes from "spawns and fails the same way" to "spawns and is best-effort"; integration coverage remains a tracked follow-up.

## Capabilities

### New Capabilities

(none — all changes live within existing capabilities)

### Modified Capabilities

- `engineer-runtime`: replace the uniform `--mcp-config` argv contract with the per-CLI `McpConfigStrategy` discriminated union; add the per-CLI argv shapes for `cursor-agent` and `codex`; add the requirement that the registry is assembled from per-CLI modules under `codingAgentClis/`.
- `role-runtime`: add the requirement that the default subprocess invoker spawns with `cwd: invocation.workspacePath` when set; replace the today-implicit "tempfile + try/finally remove" contract with an explicit `McpConfigStrategy`-driven contract whose three documented strategies cover JSON-tempfile, JSON-merge-into-workspace, and TOML-merge-into-workspace.

## Impact

- `packages/role-runtimes/src/common/codingAgentCliRegistry.ts` — keeps the abstract types, the `KnownCli` union, the `isKnownCli` guard, and the assembly of the registry constant; loses the per-CLI entry constants.
- `packages/role-runtimes/src/common/codingAgentClis/{claude,cursorAgent,codex}.ts` — three new modules, one per supported CLI, each exporting a single `CodingAgentCliEntry` constant.
- `packages/role-runtimes/src/common/codingAgentInvoker.ts` — replace `mcpConfigPathBuilder?` with `mcpConfigStrategy: McpConfigStrategy` and the strategy executor; add `cwd: invocation.workspacePath ?? undefined` to the `Deno.Command` opts.
- `packages/role-runtimes/src/common/codingAgentCliRegistry_test.ts` — extend to pin the three strategy `kind`s and the new per-CLI argv invariants; add a `workspace-json` merge test; add a `workspace-toml` round-trip test using `@std/toml`.
- `packages/role-runtimes/src/common/codingAgentInvoker_test.ts` — add coverage for the `cwd` plumbing and for the strategy executor's three branches with mocked filesystem reads / writes.
- `packages/role-runtimes/tests/integration/cursorAgent_test.ts` (new) — integration test against the real `cursor-agent` binary, gated on binary availability.
- `packages/cli/src/start/engineerRunner_test.ts` — adjust the existing test fixture's `buildArgs: () => ["--mcp-config"]` to the new strategy-aware shape so the test seam still type-checks.
- `packages/cli/src/start/engineerRunner_e2e_test.ts` — adjust the fixture entry the same way.
- `openspec/specs/engineer-runtime/spec.md` — replace the "best-effort `--mcp-config`" wording for `cursor-agent` / `codex` with the explicit strategy-driven contract.
- `openspec/specs/role-runtime/spec.md` — add the `cwd` requirement and the strategy-driven invoker contract; tighten the existing "tempfile + try/finally remove" line so it scopes only to the `tempfile-json` strategy.
- `README.md` — extend the "Configure the coding-agent CLI" section with one paragraph on what each CLI's MCP discovery contract is and where the keni MCP server entry lands.
- No SPA, REST, WS, or scheduler changes. No new third-party deps (`@std/toml` is already in Deno's std and pulled in via the workspace `deno.json` for the orchestration server's project YAML reader; if absent, this change adds it — a one-line `deno.json` edit).
