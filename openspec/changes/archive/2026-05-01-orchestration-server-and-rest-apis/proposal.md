## Why

Step 03 has shipped: `keni init` produces the on-disk `.keni/` and `~/.keni/` layouts that `spec.md` ¬ß5.1 / ¬ß5.2 prescribe, every storage interface from step 02 has a real, atomic file-backed adapter against that layout, and a project carries a stable UUIDv4 `project_id`. What is missing ‚Äî and what every later layer of Keni is structurally blocked on ‚Äî is the **orchestration server**: the local HTTP surface that turns those storage interfaces into a real service for the SPA, the role runtimes, and the user (the latter via curl in the prototype). Per `spec.md` ¬ß2#1 the environment is the communication bus; that bus is exactly this server. Per ¬ß5.3 the server is also the gatekeeper for `.keni/` writes ‚Äî every legitimate write to tickets, PRs, and the activity log on `main` flows through the API, so the server is the *only* place the ¬ß4.2 status machine ("only the owning role may transition into its own statuses") can be enforced; if the server were not the choke point, an agent could rewrite the board by editing files. Step 04 is therefore the moment the prototype becomes an actual system rather than a set of libraries: the HTTP routing skeleton lands, the REST endpoints for tickets / PRs / activity log land, the role-aware status-machine guard lands, the structured error contract lands, and a downstream `keni start` (step 13) gets a concrete program to boot. Without this step, every later step (WebSocket events in 05, MCP tools in 06, role runtimes 07‚Äì09, SPA 10‚Äì12, `keni start` 13, PO and chat 14‚Äì15) would be testable only against in-memory mocks; with it, every later step gets a real HTTP service to drive against.

## What Changes

- Introduce a new `@keni/server` HTTP orchestration server, replacing the placeholder `packages/server/src/main.ts` (which today only exports `packageName`) with a real runnable program. The server is local-only in the prototype (no auth, binds to `127.0.0.1` by default), runs against a single project (the one whose `.keni/` lives at the configured project root, per `spec.md` ¬ß7.1's "one server instance manages one project"), and is started either programmatically (`createServer(deps)` returning a Hono app) or via a thin `startServer(opts)` helper that wraps `Deno.serve` and returns an `AbortController` so step 13's `keni start` and integration tests can lifecycle it deterministically.
- Pick **Hono** (`jsr:@hono/hono`) as the server framework. Justification lives in `design.md` (Decision 1): it is the smallest first-class router for Deno, has zero transitive runtime deps, ships JSR-native, gives us typed `c.req.json<T>()` / `c.json(body, status)` ergonomics, and supports the middleware shape we need for request logging and the role-guard. The alternatives (hand-rolled `Deno.serve` router; Fastify / Express via Node compat) either leak boilerplate into every handler or pull in npm Node-emulation surface area.
- Pick **zod** (`npm:zod`) as the validation library, with all schemas living under `packages/server/src/wire/` and the corresponding wire-shape TypeScript types living under `packages/shared/src/wire/`. The server consumes the zod schemas at the request boundary; the SPA (and any other typed consumer) imports only the inferred TS types from `@keni/shared` and tree-shakes zod out of its bundle. A compile-time `z.ZodType<TicketCreateRequest>` constraint per schema enforces that the schema and the shared type stay aligned. Justification and the alternative (typebox / valibot / hand-rolled JSON schema) live in `design.md` (Decision 5).
- Define a wire-shape contract in `@keni/shared/wire/` covering tickets, PRs, and activity log: request shapes (`TicketCreateRequest`, `TicketHeaderPatchRequest`, `TicketTransitionRequest`, the PR equivalents, `ActivityAppendRequest`, `ActivityQueryRequest`), response shapes (`TicketResponse`, `TicketListResponse`, `PRResponse`, `ActivityEntryResponse`, `ActivityQueryResponse`), and the structured error envelope (`ErrorResponse`). Every shape carries `project_id` (read-only, server-populated from the resolved project config) so a future multi-project server is purely additive (`spec.md` ¬ß7.1 and ¬ß11 #5 both promise this). The shapes are deliberately *separate* from the storage record types: storage records have on-disk fields (`created_at`, `updated_at`, file path implied by id); wire records have HATEOAS-friendly fields (`self`, request-scoped `project_id`) and a stable JSON casing the SPA can render.
- Implement the **REST endpoints for tickets**:
  - `GET /tickets` ‚Äî list, with optional filters via query string (`?status=open,in_progress`, `?assignee=alice`, `?assignee=null`, `?priorityMin=0&priorityMax=100`, `?changeRequest=cr-0007`). Server delegates to `TicketStore.list(filter)`. Returns `200 { data: TicketSummary[], project_id }`.
  - `GET /tickets/:id` ‚Äî read one. Server delegates to `TicketStore.read(id)`. Returns `200 { data: TicketResponse, project_id }`. `StoreNotFoundError` ‚Üí `404`.
  - `POST /tickets` ‚Äî create. User-driven in the prototype (the SPA acts as the `user` role; a curl call with `X-Keni-Role: user` is equivalent). Body: `TicketCreateRequest` (title, optional body, optional assignee, priority, optional change_request). Server delegates to `TicketStore.create(...)`. Returns `201 { data: TicketResponse, project_id }`. Validation failure ‚Üí `400`.
  - `PATCH /tickets/:id` ‚Äî update header (title / assignee / priority / change_request) and/or body. Body: `TicketHeaderPatchRequest` (header fields all optional, plus optional `body`). Status changes are explicitly rejected (the storage layer already throws `InvalidArtifactError("status_in_patch")`; the server propagates that as `400` with a stable error code so the SPA can surface a clear message). Returns `200 { data: TicketResponse, project_id }`.
  - `POST /tickets/:id/transition` ‚Äî transition status. **Status-machine enforcement lives here.** Body: `TicketTransitionRequest { from: TicketStatus, to: TicketStatus }`. Server validates: (a) the `to` status is reachable from the current status under the ¬ß4.1 graph, (b) the calling role (read from `X-Keni-Role`) is the owning role for `to` per ¬ß4.2 (engineer for `in_progress` ‚Üí `merged`; QA for `in_testing`, `tested`, `test_failed`; PO for `done`; `user` permitted for *any* transition with the override mark, but the override-confirmation flow itself lives in step 25 ‚Äî the prototype wires the role guard and leaves a clearly-marked TODO for the override path so the role guard is never silently bypassed). Then delegates to `TicketStore.transitionStatus(id, from, to)`. `StaleStateError` (server's `from` differed from the on-disk status) ‚Üí `409`. Disallowed-by-graph or disallowed-by-role ‚Üí `403` with a structured `error.code` (`status_graph_violation` / `role_not_owner`). Returns `200 { data: TicketResponse, project_id }`.
- Implement the **REST endpoints for PRs**, mirroring the ticket surface:
  - `GET /prs` ‚Äî list with filters (`?status=`, `?ticket=`, `?author=`).
  - `GET /prs/:id` ‚Äî read one.
  - `POST /prs` ‚Äî create (engineer-authored in the prototype; role guard accepts `engineer` and `user`).
  - `PATCH /prs/:id/intent` ‚Äî replace intent body (`PRStore.updateIntent`).
  - `POST /prs/:id/transition` ‚Äî transition status, same role guard model as tickets (engineer owns the entire PR lifecycle per `spec.md` ¬ß3 / ¬ß4.1; the user can override).
- Implement the **REST endpoints for the activity log**:
  - `GET /activity` ‚Äî query with filters (`?agent=`, `?role=`, `?from=`, `?to=`). The store returns an `AsyncIterable`; the server materialises it into a single page response in the prototype (no pagination for now ‚Äî the dataset is bounded by date partitioning and the activity directory is brand-new on a fresh project; pagination is an additive change in a later step if a long-running project produces multi-megabyte days). Returns `200 { data: ActivityEntryResponse[], project_id }`.
  - `POST /activity` ‚Äî append entry. Body: `ActivityAppendRequest` (timestamp optional, session_id, agent, role, event, summary, refs). Returns `201 { data: ActivityEntryResponse, project_id }`. `InvalidArtifactError("size_exceeded")` ‚Üí `422`.
- Add the **role-identity contract**: every request carries `X-Keni-Role` (one of `user`, `engineer`, `qa`, `po`, `writer`) and optionally `X-Keni-Agent` (the agent id, for activity-log attribution and future multi-engineer guards in step 26). The header is *trusted* in the prototype ‚Äî there is no auth, the server runs on `127.0.0.1`, the role is a soft contract between the SPA / role runtimes / curl and the server. The middleware that reads the header populates a typed `Context.var.role` / `Context.var.agent` that handlers consume. Missing or unknown role ‚Üí `400` with `error.code: "missing_role"` so misconfigured callers fail loudly rather than silently bypassing the role guard. The contract is documented in `design.md` (Decision 4) and the `orchestration-server` capability spec.
- Add a **structured error contract**: every non-2xx response returns `{ error: { code, message, details? } }` with a stable `error.code` set documented in the capability spec (`store_not_found`, `stale_state`, `duplicate_id`, `invalid_artifact`, `status_graph_violation`, `role_not_owner`, `missing_role`, `validation_failed`, `internal_error`). A single error-mapping function (`packages/server/src/errors.ts`) maps storage errors and the new server-domain errors (`StatusGraphViolationError`, `RoleNotOwnerError`, `MissingRoleError`) onto the `(httpStatus, errorBody)` pair so handlers never construct ad-hoc error responses inline.
- Add **request logging middleware**: one structured JSONL line per request, written to stdout in development and to `~/.keni/logs/server-YYYY-MM-DD.jsonl` when running under `keni start` (the file destination is wired here because step 13 already gets the `~/.keni/logs/` directory from `keni init` per the project-layout spec; step 13 just opts into the file-sink config). Fields: `request_id` (per-request UUIDv4), `method`, `path`, `status`, `duration_ms`, `role`, `agent`, `project_id`, plus `error.code` on non-2xx. The middleware also stamps `X-Keni-Request-Id` on the response so curl / SPA logs line up.
- Add a **project-id resolution layer**: on server bootstrap, `createServer({ configStore })` calls `configStore.readProjectConfig()` once and caches the resolved `project_id` for the life of the process. Every response carries that id; every request that names a project id (none today, but the wire shapes anticipate multi-project) validates it. Resolution failure (no `project.yaml`) is a fatal startup error with a clear message ("run `keni init` first"). This is the seam step 13 and a future multi-project step (post-MVP) build on.
- Wire-up integration tests: end-to-end tests against the real server (built via `app.fetch(new Request(...))` against the Hono app, no port binding required ‚Äî the same pattern Hono itself documents) covering every endpoint plus the role-guard, status-graph, error-mapping, and request-logging contracts. Tests run in `Deno.makeTempDir()`, instantiate the file-backed stores against that dir, and exercise the full HTTP path.
- A new `orchestration-server` capability spec captures the API contract end-to-end: every endpoint's method/path/request-shape/response-shape/role-owner, the structured error envelope and the stable error-code set, the role-identity header contract, the project-id resolution rule, the status-graph definition (the ¬ß4.1 transitions encoded as a server-owned constant), the request-logging shape, and the boundary between this capability (HTTP gateway, role guard, status-graph enforcement, error mapping, request logging) and `storage` (the on-disk shape and atomicity it builds on). Step 05 (WebSocket events + `/agents`) and step 06 (MCP tools) will both add deltas to this spec.
- Update the root `README.md` "Getting started" section with a short note on running the server in dev mode (`deno run -A packages/server/src/main.ts --project=<path>`) and a one-line `curl` example against `/tickets` so the contract is documented for readers, with a forward reference that step 13 will fold this into `keni start`.

## Capabilities

### New Capabilities

- `orchestration-server`: The HTTP REST API contract for Keni's local orchestration server. Names every endpoint (method, path, request shape, response shape, status codes), the structured error envelope (`{ error: { code, message, details? } }`) and the stable error-code set, the role-identity header contract (`X-Keni-Role`, `X-Keni-Agent`), the status-graph constant (the ¬ß4.1 transitions encoded server-side), the role-owner mapping per ¬ß4.2 (which role may transition into which status), the project-id resolution rule (one server, one project, id read once at bootstrap from `.keni/project.yaml`), the request-logging line shape, and the structural boundary between the server (gateway, guard, mapper, logger) and the `storage` capability (on-disk shape and atomicity). This capability does **not** describe how individual files are written (that is `storage`) or how the server is started from the CLI (that is the future `keni start` capability owned by step 13); it describes only the HTTP contract that the SPA, role runtimes, MCP layer, and curl all bind to. Step 05 (WebSocket events and `/agents` endpoint) and step 06 (MCP tools) will both add additive deltas to this spec.

### Modified Capabilities

None. The `storage` interfaces (step 02) and the `project-layout` contract (step 03) are consumed without modification ‚Äî this step is purely additive. No existing requirement in any prior capability changes.

## Impact

- **Affected code**:
  - `packages/server/src/main.ts` is rewritten from a placeholder constant to the real server entry point: `createServer(deps): Hono` (composition root), `startServer(opts): { abort, port, url }` (process-level lifecycle wrapper), `runServer(args): Promise<number>` (CLI-style entry that returns an exit code, used by step 13).
  - `packages/server/src/middleware/` (new): `requestId.ts`, `requestLog.ts`, `roleIdentity.ts`, `errorBoundary.ts`. Each is a typed Hono middleware factory accepting its dependencies (e.g. `roleIdentity` accepts no deps; `requestLog` accepts a `LogSink` interface) so unit tests can swap fakes.
  - `packages/server/src/routes/` (new): `tickets.ts`, `prs.ts`, `activity.ts`. Each exports a Hono route group bound to its store interface. Handlers are thin: parse ‚Üí validate via zod ‚Üí role/graph guard ‚Üí store call ‚Üí map storage record to wire shape ‚Üí respond.
  - `packages/server/src/wire/` (new): zod schemas for every request and response shape. Each schema is exported alongside its `z.ZodType<SharedTypeFromShared>` constraint so a drift between the server-side schema and the shared TS type fails the type-check at compile time.
  - `packages/server/src/errors.ts` (new): the new `StatusGraphViolationError`, `RoleNotOwnerError`, `MissingRoleError` classes plus the central `mapErrorToResponse(err): { status, body }` function. Storage errors (`StoreNotFoundError`, `StaleStateError`, `DuplicateIdError`, `InvalidArtifactError`) are imported from `@keni/shared` and mapped here without modification.
  - `packages/server/src/statusGraph.ts` (new): the ¬ß4.1 ticket-status graph and the ¬ß4.2 role-ownership map, as exported constants (frozen object literals). Reused by the `transition` handlers, by the integration tests, and (in a later step) by the SPA via `@keni/shared` re-exports if useful ‚Äî for now it stays server-internal.
  - `packages/server/src/main_test.ts` is extended from a placeholder to a small smoke test that imports `createServer` and `app.fetch(new Request("/tickets"))` returns 200 in a temp dir.
  - `packages/server/src/routes/*_test.ts` (new, one per route file): end-to-end tests against the Hono app via `app.fetch`, covering each endpoint plus role-guard, status-graph, validation, and error-mapping.
  - `packages/server/src/middleware/*_test.ts` (new): unit tests for each middleware (request-id assignment, log-line shape, role-header parsing including the missing/unknown cases, error-boundary mapping).
  - `packages/shared/src/wire/` (new): TypeScript types for every request and response shape, plus the `ErrorResponse` envelope. No zod, no runtime code ‚Äî types only. Re-exported from `@keni/shared`.
  - `packages/shared/src/main.ts` adds `export * from "./wire/mod.ts"` so consumers can `import type { TicketCreateRequest } from "@keni/shared"`.
  - `packages/shared/src/storage/README.md`: a one-paragraph note pointing readers at the new wire-shape module and clarifying the storage / wire boundary.
  - Root `README.md`: short "Run the orchestration server" subsection in "Getting started", with the dev invocation and a curl example, plus the forward reference to step 13.
  - Root `deno.json`: add `jsr:@hono/hono@^4` and `npm:zod@^3.23` to the imports map (pinning to the latest stable major; lock file gets regenerated). No other workspace member's `deno.json` needs touching ‚Äî Hono and zod are server-internal except for the `import type` consumption from `@keni/shared/wire/`, which has no runtime cost.
- **Affected APIs / contracts**:
  - **Public (HTTP)**: every endpoint listed under "What Changes" ‚Äî methods, paths, request and response shapes, status codes, error codes, role-identity headers. This is the SPA's and the role-runtimes' contract for steps 07+ and 10+.
  - **Public (TS, via `@keni/shared/wire`)**: every wire-shape type. The SPA and role runtimes import these as types only.
  - **Internal**: `createServer(deps)`, `startServer(opts)`, `runServer(args)` from `@keni/server`. Step 13 imports these.
  - **No change** to: `TicketStore`, `PRStore`, `ActivityLogStore`, `ConfigStore` interfaces; the on-disk shapes under `.keni/`; the storage error classes; the id-generation module; the `keni init` CLI; the `.keni/` / `~/.keni/` layout.
- **Affected dependencies**:
  - **New runtime dependencies**: `jsr:@hono/hono@^4` (server framework, ~14 KB minified, zero transitive deps in Deno), `npm:zod@^3.23` (validation library, single dep, well-supported on Deno via the npm: specifier). Both pinned in the root `deno.json` and locked in `deno.lock`.
  - **No new dev dependencies**. Hono's testing pattern (`app.fetch(new Request(...))`) needs no extra tooling; zod's test pattern is plain `schema.parse(...)` / `schema.safeParse(...)`.
  - **No change** to: the `@std/*` set (already covers paths, fs, uuid, yaml, assert), Deno's runtime version pin, the CI configuration.
- **Affected tests**:
  - **New**: route integration tests under `packages/server/src/routes/` (one file per route group, each covering the happy path, every storage-error mapping, every role-guard refusal, every status-graph violation, and every validation refusal). Middleware unit tests under `packages/server/src/middleware/`. Wire-schema unit tests under `packages/server/src/wire/` asserting the schemas accept and reject the documented examples.
  - **Extended**: `packages/server/src/main_test.ts` gains a smoke test calling `createServer` and `app.fetch("/tickets")` against a temp-dir-backed `FileTicketStore` ‚Äî proves the composition root wires the deps correctly.
  - **No change**: the storage capability's 300+ existing tests (step 03 final count) all stay green.
- **Downstream steps unblocked**:
  - **Step 05 (WebSocket events + `/agents` endpoint)** gets a real Hono app to attach the WS upgrade handler to and the `/agents` endpoint to mount alongside `/tickets`.
  - **Step 06 (MCP tools)** gets the same store-binding pattern (composition root passes interfaces, MCP tools call interface methods, status guard belongs to the same `statusGraph.ts` module the server imports ‚Äî MCP can re-use it).
  - **Steps 07‚Äì09 (role runtimes)** get a real HTTP API to call: tickets, PRs, activity log all reachable, role-guard enforced, error contract stable.
  - **Steps 10‚Äì12 (SPA)** get the wire-shape types from `@keni/shared` and the HTTP API to call.
  - **Step 13 (`keni start`)** gets a `runServer(args)` entry to dispatch to from the CLI, plus the request-log file destination already pointed at `~/.keni/logs/`.
  - **Step 15 (chat endpoints)** mounts on the same Hono app additively.
  - **Step 25 (manual override)** adds the override-confirmation flow to the existing `transition` endpoints; the role-guard already routes the user role through, the override flow just adds the confirmation check and the `manual_override` activity-log entry.
- **Non-impact (deliberate)**:
  - **No WebSocket**, **no `/agents` endpoint** ‚Äî step 05 owns those. The server is HTTP-only in this step.
  - **No MCP layer** ‚Äî step 06 owns that. The MCP server is a separate process that talks to the orchestration server (or, equivalently, to the same store interfaces); this step does not pre-shape that boundary.
  - **No chat endpoints** ‚Äî step 15 owns those.
  - **No auth, no TLS, no CORS** ‚Äî the prototype binds to `127.0.0.1` and trusts the role header. Step 22 (or wherever auth lands post-MVP) introduces an auth layer additively, in front of the role guard.
  - **No `manual_override` event logging in this step** ‚Äî the `transition` endpoint enforces the role guard but does not emit `manual_override` activity entries when the user role overrides; that confirmation-and-log flow is step 25. The server leaves a clearly-marked TODO and a corresponding spec scenario marking the gap.
  - **No `keni start` integration** ‚Äî step 13 owns the CLI dispatch. This step ships a programmatic entry point and a developer-only `deno run` invocation; nothing in `packages/cli/` changes.
  - **No multi-project support** ‚Äî the server is one-server-one-project per `spec.md` ¬ß7.1. Wire shapes carry `project_id` as read-only / server-populated so a multi-project server is purely additive (the same shapes will accept `project_id` as a request-scoped field then), but the prototype rejects any inbound `project_id` that does not match the resolved one.
  - **No pagination on `GET /activity`** ‚Äî bounded prototype data; pagination is additive when needed.
  - **No request-rate limiting**, **no request-size cap beyond what zod imposes per field** ‚Äî local-only server, single user.
  - **No prompt embedding work** ‚Äî prompts ship per `spec.md` ¬ß11#3 and the `developer-setup` convention; this step does not touch them.

## Spec references

- ¬ß2#1 ‚Äî Environment as communication bus. The HTTP server *is* that bus for the SPA, role runtimes, MCP, and the user.
- ¬ß2#3 ‚Äî Status drives behaviour. Transitions are the only legitimate state changes; the server enforces this in one place.
- ¬ß4.1 ‚Äî Ticket lifecycle. Every status the `transition` endpoint must support is enumerated here; the server-side graph constant tracks it 1:1.
- ¬ß4.2 ‚Äî Owning-role rule, `test_failed` as an explicit status, no `rejected` status, priority is PO-owned integer. The server enforces the owning-role rule; the absence of `rejected` is encoded in the graph.
- ¬ß4.3 ‚Äî Who creates tickets. In the prototype the user creates directly; the API accepts user-authored tickets (`X-Keni-Role: user`).
- ¬ß5.3 ‚Äî Server is the gatekeeper for `.keni/` writes. The API is the only legitimate writer for tickets / PRs / activity log on `main`. This is the architectural reason the role guard cannot live elsewhere.
- ¬ß7.1 ‚Äî One server instance manages one project; `keni start` boots this server (step 13). Every artifact carries `project_id` so a future multi-project server is additive.
- ¬ß11#5 ‚Äî Files first, storage abstracted. The server consumes the storage interfaces, never `Deno.readTextFile` against `.keni/`.
- ¬ß11#10 ‚Äî Engineer workspaces are code-only; metadata is API-managed. The server is the API.
