## Why

Steps 04 and 05 turned `.keni/` into a real local service: the orchestration server owns the only legitimate write path to tickets, PRs, and the activity log on `main`, enforces the §4.1 status graph and the §4.2 owning-role rule on every transition, and broadcasts every successful write over a typed event stream. What is missing — and what every later piece of the prototype's autonomous loop is structurally blocked on — is the **engineer's API surface**: a coding-agent subprocess (Claude Code, Cursor agent, OpenCode, etc.) cannot drive the board through `curl`, cannot be trusted to fabricate role headers correctly, and **must not** see `.keni/` (the workspace clone is sparse precisely so the engineer cannot edit metadata files; `spec.md` §5.3). `spec.md` §5.4 names the way out: "All other reads and writes happen through MCP tools exposed by Keni." Step 06 ships exactly that — a stdio MCP server the role runtime hands to the engineer subprocess so the agent has a small, audited, role-attributed verb set (list / read / update tickets, append / query the activity log, resolve its workspace path) and **only** that. Without this step, every later step (role runtime in 07, scheduler in 08, engineer specialisation in 09, end-to-end smoke in 13) has no API the subprocess can call and the `spec.md` §5.3 write boundary becomes a pinky-promise rather than an enforced contract.

This step also fixes the architectural tension §6.4 ("agent-runner agnosticism") would otherwise create. The whole point of the thin-runtime pattern is that the coding agent is interchangeable — Claude Code today, Cursor agent tomorrow — and the only seam that makes that safe is the MCP tool list. Each supported coding-agent CLI already speaks MCP natively (stdio + JSON-RPC); each one will run the same tools. Without the MCP server, "swap the coding agent" turns into "rewrite the role-runtime's prompt against whatever ad-hoc REST hack the new CLI prefers" — exactly the surface §11#3 ("prompts as code, not files") and §2#4 ("thin wrapper, agentic decisions") were drawn to avoid.

## What Changes

- Stand up a new **stdio MCP server** under `packages/server/src/mcp/`, runnable as `deno run -A packages/server/src/mcp/main.ts --agent <id> --server-url <url> --workspace <path>`. The server is **its own process**, spawned by the role runtime as part of the engineer subprocess's MCP configuration (every supported coding-agent CLI accepts an `mcpServers` JSON block whose entries point at command + args + env), and communicates with the engineer subprocess over stdio via JSON-RPC per the MCP spec. The MCP server itself is a thin client of the orchestration server's REST surface from steps 04 and 05 — every tool delegates to a typed HTTP call against `http://127.0.0.1:<port>` carrying `X-Keni-Role: engineer` and `X-Keni-Agent: <id>`. No tool reads or writes `.keni/` directly; no tool re-implements status-graph, role-owner, or atomicity logic. This is the load-bearing decision: the MCP layer is *a transport adapter*, not a duplicate of the API.
- Pick **the official `@modelcontextprotocol/sdk` (v1.x, `npm:@modelcontextprotocol/sdk@^1`)** as the MCP server framework. Justification lives in `design.md` (Decision 1): it is the only first-party TypeScript SDK, ships zod v4 schema bindings (matching the project's pinned `npm:zod@^4`), runs on Deno via the `npm:` specifier per its README, and exposes the exact `McpServer` + `StdioServerTransport` primitives the input file calls for. The alternatives (hand-rolling JSON-RPC over stdio; the `@hono/mcp` middleware path) either duplicate hundreds of lines of protocol bookkeeping or force an HTTP transport the input file deliberately rejects in favour of stdio.
- Expose **seven focused tools** (one verb per tool, deliberately no aggregate `update_ticket`) — names, scopes, and contracts:
  - **Tickets**: `list_tickets` (delegates to `GET /tickets`, accepts the same `status`, `assignee`, `priorityMin`, `priorityMax`, `change_request` filters), `read_ticket` (`GET /tickets/:id`), `update_ticket_body` (`PATCH /tickets/:id` carrying only the `body` field; the storage layer's `status_in_patch` guard already rejects any sneaked-in `status`), `transition_ticket_status` (`POST /tickets/:id/transition`, body `{ from, to }` — the orchestration server's existing `TICKET_STATUS_TRANSITIONS` graph and the engineer-only owning-role rows in `TICKET_STATUS_OWNING_ROLES` enforce reachability and ownership; the MCP server adds nothing here).
  - **Activity log**: `append_activity_entry` (delegates to `POST /activity`, accepts `event`, `summary`, `refs`, `session_id`; `agent` and `role` are stamped server-side from the MCP process's identity rather than re-passed by the agent — see Decision 4 in `design.md`), `query_activity` (delegates to `GET /activity`, accepts `agent`, `role`, `from`, `to`, plus a tool-level `limit` capped at 200 entries by default to keep an LLM context window from being burned by a chatty day partition).
  - **Workspace**: `get_workspace_path` — returns `{ path: string }` where `path` is the absolute filesystem path of the engineer's workspace clone. The path is resolved at MCP-server bootstrap from the `--workspace <path>` CLI argument the role runtime injects (step 07 owns workspace provisioning; step 09 wires this for engineers specifically) and stays constant for the lifetime of the MCP-server process — the value is read once and stamped on every call so the agent cannot trick the tool into returning a different directory mid-session.
- Define the **identity-propagation contract**: every MCP-server invocation receives `--agent <id>` and `--server-url <http url>` (and `--workspace <abs path>`); these are validated at startup; the server refuses to start (exit code 2, message on stderr) when any required argument is missing or malformed. Every outbound HTTP call from a tool handler then carries `X-Keni-Role: engineer` and `X-Keni-Agent: <id>` (using the existing role-header trust model from step 04 — local-only, no auth, headers trusted). The MCP server **never** lets the agent override the agent id or the role through tool input; `agent_id` is not even a tool parameter. This is the only place the §4.2 owning-role rule shows up in the MCP layer: it cannot be bypassed because it is bypassed nowhere — the MCP server stamps the headers from its own boot-time configuration, and the orchestration server's existing `roleIdentity` middleware does the actual validation.
- Define the **error contract** for tool calls: the MCP SDK's `isError: true` return shape is the surface; every tool wraps its HTTP call in a small adapter that maps the orchestration server's `{ error: { code, message, details? } }` envelope into the MCP `content: [{ type: "text", text: <human-readable> }]` payload plus a structured `isError: true` flag. The closed `ErrorCode` enum from step 04 is reused verbatim (`store_not_found`, `stale_state`, `status_graph_violation`, `role_not_owner`, `validation_failed`, etc.) — the agent gets the same vocabulary as the SPA. No new error codes. Network-level failures (orchestration server unreachable, connection reset) map to a tool error tagged `internal_error` with a message naming the URL the call targeted, so the role runtime's activity-log capture surfaces it cleanly.
- Wire **integration tests** that prove the end-to-end loop: a temporary `keni init`-produced project, the orchestration server running on a random port, the MCP server spawned with `--agent alice --server-url http://127.0.0.1:<port> --workspace <tempdir>`, and an in-process MCP **client** (`@modelcontextprotocol/sdk/client`) that connects via the SDK's `StdioClientTransport`, lists the seven tools, and exercises each end-to-end — list, read, update, transition (one engineer-allowed `from→to` and one `role_not_owner` refusal as a negative test), append + query activity, get workspace path. The fake-agent harness lives at `packages/server/src/mcp/integration_test.ts` and is the load-bearing safety net: it asserts that the role-owner / status-graph / atomicity guarantees that step 04 already enforces remain enforced when the call comes through MCP rather than direct HTTP.
- Define the **`mcp-engineer-surface` capability spec**: every tool with parameters, return shape, error mapping, idempotency, and which orchestration-server endpoint it delegates to. The spec is the document step 09 (engineer prompt) reads to write the engineer's system prompt and the document step 26 (multi-engineer) reads when it adds per-agent guards. It also names the explicit non-goals (no PO tools, no spec / CR tools, no PR-write tools, no WS tools — those are §5.3 PO-direct or step 16 / 09 territory) so a later contributor cannot accidentally extend this surface without an OpenSpec change.
- Update the root `README.md` "Run the orchestration server" subsection with a short forward-reference to the MCP server: how the role runtime will spawn it (step 07), where the entry point lives, and the one-line `deno run` invocation a developer can use to attach a manual MCP client (e.g. `mcp-cli` or `claude-code mcp` style debugging) for a sanity check before step 09 lands.
- Add `npm:@modelcontextprotocol/sdk@^1` to the workspace `deno.json` imports map, regenerate `deno.lock`. No other workspace member imports it — the SDK lives entirely under `packages/server/src/mcp/` and is internal to that subtree.

## Capabilities

### New Capabilities

- `mcp-engineer-surface`: The contract for the engineer's MCP tool surface. Names every tool (`list_tickets`, `read_ticket`, `update_ticket_body`, `transition_ticket_status`, `append_activity_entry`, `query_activity`, `get_workspace_path`), its parameter schema (zod v4 against the existing `@keni/shared/wire/` shapes), its return shape, the orchestration-server endpoint it delegates to, and the error mapping from the REST `ErrorCode` envelope into the MCP `isError: true` shape. Names the identity-propagation contract (the MCP server is spawned with `--agent <id>` / `--server-url <url>` / `--workspace <path>`; it stamps `X-Keni-Role: engineer` and `X-Keni-Agent: <id>` on every outbound HTTP call; the agent cannot override these via tool input). Names the transport choice (stdio + JSON-RPC per the MCP spec, hosted by `@modelcontextprotocol/sdk`'s `StdioServerTransport`). Names the explicit out-of-scope list — PO chat / CR / spec tools (step 16), PR record CRUD (engineers create PRs through the role runtime's git handling, step 09), `.keni/de-facto-spec/` and `.keni/changes/` (PO-direct per §5.3), WebSocket-style streaming tools (MCP is request/response). Names what survives multi-engineer (step 26): the per-tool role-owner guard sits inside the orchestration server, so the MCP server itself does not need a per-agent code path — adding more engineers is just spawning more MCP-server processes with different `--agent` values.

### Modified Capabilities

None. The `orchestration-server` capability (steps 04 + 05) is consumed without modification — every MCP tool delegates to an existing endpoint over the existing wire contract, with the existing role-identity middleware doing the validation. No requirement is changed, weakened, or removed; the closed `ErrorCode` enum is preserved verbatim. The `storage` and `project-layout` capabilities are likewise unchanged.

## Impact

- **Affected code**:
  - `packages/server/src/mcp/main.ts` (new): runnable entry point — exports `runMcpServer(args): Promise<number>`; calls `Deno.exit(...)` only under `import.meta.main`. Mirrors the orchestration server's `main.ts` shape from step 04.
  - `packages/server/src/mcp/createMcpServer.ts` (new): pure factory — given a typed deps object (`{ httpClient, agentId, workspacePath }`) returns a fully-configured `McpServer` with all seven tools registered. Synchronous, no I/O, easy to unit-test.
  - `packages/server/src/mcp/runMcpServer.ts` (new): CLI entry — parses argv (`--agent`, `--server-url`, `--workspace`), validates, instantiates the HTTP client, calls `createMcpServer(...)`, connects a `StdioServerTransport`, and waits for shutdown. Returns 0 on clean exit, 1 on runtime failure, 2 on usage error.
  - `packages/server/src/mcp/httpClient.ts` (new): the typed HTTP adapter. Exports `createMcpHttpClient(opts: { serverUrl, agentId })` returning an object with one method per delegated endpoint (`listTickets`, `readTicket`, `updateTicketBody`, `transitionTicket`, `appendActivity`, `queryActivity`). Each method serialises arguments, stamps `X-Keni-Role: engineer` + `X-Keni-Agent: <id>`, parses the response envelope, and throws a typed `McpHttpError(code, message, details?)` on non-2xx so the tool layer can map it cleanly.
  - `packages/server/src/mcp/tools/` (new): one file per tool group — `tickets.ts` (four tools), `activity.ts` (two tools), `workspace.ts` (one tool). Each file exports a `register<Group>Tools(server, deps)` function that calls `server.registerTool(...)` for each tool with its zod schema and handler.
  - `packages/server/src/mcp/errors.ts` (new): `McpHttpError` class and `mapHttpErrorToToolResult(err): { content, isError: true }` — the central function every tool handler funnels failures through.
  - `packages/server/src/mcp/main_test.ts`, `createMcpServer_test.ts`, `runMcpServer_test.ts`, `httpClient_test.ts`, `tools/<group>_test.ts`, `errors_test.ts` (new): unit tests per layer, mirroring the orchestration server's test layout.
  - `packages/server/src/mcp/integration_test.ts` (new): the end-to-end test harness — spins up a real orchestration server on a random port, spawns the MCP server as a `Deno.Command` subprocess with stdio piped, attaches an in-process MCP `Client` over the SDK's `StdioClientTransport`, calls every tool, asserts on side effects (file changes under `.keni/tickets/`, activity-log entries on disk, role-owner refusals, status-graph refusals).
  - `packages/server/src/mcp/wire/` (new): zod v4 input schemas for the tool parameters, expressed as `z.ZodType<SharedTypeFromShared>` against the existing `@keni/shared/wire/` ticket / activity / role types. Reuses the same drift-detection pattern step 04 introduced.
  - `packages/server/src/main.ts` (modified): re-exports `runMcpServer` and the public types (`McpServerDeps`, `McpHttpClient`) so step 07's role runtime can import them without reaching into a subdirectory. No change to the orchestration server's existing exports.
  - `packages/shared/src/wire/mcp.ts` (new, *optional* — see `design.md` Open Question 1): the typed return-shape interface for `get_workspace_path` (`WorkspacePathResponse = { path: string }`) lives here so the role runtime (step 07) and the SPA (later, if it ever wants to mirror "where does alice work") can read it. If we decide the type is internal to `@keni/server/mcp/`, the file lives there instead. Decision pinned in `design.md`.
  - `deno.json` (modified): adds `npm:@modelcontextprotocol/sdk@^1` to the imports map under the alias `@modelcontextprotocol/sdk`. `deno.lock` regenerated.
  - `README.md` (modified): "Run the orchestration server" subsection extended with a forward-reference to the MCP server invocation and a note that step 07 is what wires it into the engineer subprocess.
- **Affected APIs / contracts**:
  - **Public (MCP, JSON-RPC over stdio)**: the seven tools, their parameter schemas, their return shapes, their error envelope. This is the engineer subprocess's contract.
  - **Public (TS, via `@keni/server`)**: `runMcpServer(args)`, `McpServerDeps`, `McpHttpClient`. Step 07 imports these.
  - **Internal**: every file under `packages/server/src/mcp/` is internal to `@keni/server`; no other workspace package imports from this subtree.
  - **No change** to: the orchestration server's HTTP surface (`/tickets`, `/prs`, `/activity`, `/agents`, `/events`); the storage interfaces; the on-disk `.keni/` layout; the wire shapes for tickets / PRs / activity / agents / events / errors / role; the `ErrorCode` enum; the `EventBus` / `AgentRuntimeStateStore` contracts; the role-identity middleware; the `keni init` CLI; `~/.keni/` layout.
- **Affected dependencies**:
  - **New runtime dependency**: `npm:@modelcontextprotocol/sdk@^1` (the official MCP TypeScript SDK; v1.29.0 is the current stable line per the project's `today` reference; ESM-first, runs on Deno via the `npm:` specifier). Single transitive dep tree dominated by the SDK itself; one zod-v4 binding (already in `deno.json`); one `eventsource-parser` for HTTP transport which we do not use (we use stdio only — Deno tree-shakes by import path). Pinned `^1` in `deno.json`, locked in `deno.lock`.
  - **No new dev dependencies**. The SDK ships its own client transport (`StdioClientTransport`) which the integration test imports.
  - **No change** to: `@hono/hono`, `zod`, `@std/*` already in `deno.json`. The orchestration server's deps are untouched.
- **Affected tests**:
  - **New**: ~7 unit-test files under `packages/server/src/mcp/` (per-layer, ~5 tests each) + 1 integration-test file exercising every tool end-to-end through a real subprocess (~10 tests). Estimated ~50 new tests.
  - **No change**: every existing test in the orchestration server, storage layer, CLI, or SPA stays green. The new tests are entirely additive.
- **Downstream steps unblocked**:
  - **Step 07 (role runtime)** gets `runMcpServer` to point at when wiring the engineer subprocess's `mcpServers` config. The role runtime instantiates the MCP-server invocation (a `Deno.Command` config with `--agent <id> --server-url <url> --workspace <path>`); the coding-agent CLI consumes that config, spawns the MCP server, and the loop closes.
  - **Step 09 (engineer specialisation)** gets the typed tool list to write the engineer's system prompt against. The prompt names the seven tools by their MCP names; the prompt is `prompts as code` (per §11#3) and lives next to the role runtime, not as a file under `.keni/`.
  - **Step 13 (`keni start`)** ships unchanged. The MCP server is spawned per-engineer-cycle by the role runtime, not by `keni start`.
  - **Step 16 (PO MCP additions)** layers PO-only tools (`chat_send`, `ticket_create_from_cr`, `pr_read`, etc.) onto the same `McpServer` factory pattern and the same identity-propagation contract — the per-tool registration is the seam that keeps PO tools out of the engineer surface.
  - **Step 26 (multi-engineer)** spawns more MCP-server processes (one per engineer) without changing the MCP server itself: the per-agent identity comes from the `--agent` CLI flag, and the orchestration server already enforces the per-agent guard.
- **Non-impact (deliberate)**:
  - **No HTTP transport for MCP.** Stdio only, per the input file's "Pick stdio for now and document." HTTP is a §10 future-additive extension; the `createMcpServer` factory's transport is injected so swapping it is one line.
  - **No new endpoints on the orchestration server.** Every tool delegates to an existing endpoint. If a future tool needs something the REST surface does not yet expose, that surface lands in the relevant capability spec first; the MCP layer never grows its own back door.
  - **No PO-specific tools.** Step 16 owns those (`chat_send`, `ticket_create_from_cr`, `pr_read`, etc.). The MCP server in this step is engineer-scoped — every tool registration check the role argument matches `engineer` at startup and refuses to start under any other role (so a contributor cannot accidentally hand the engineer surface to a PO subprocess).
  - **No `.keni/de-facto-spec/` or `.keni/changes/` tools.** Those are PO-direct per §5.3.
  - **No PR-write MCP tools.** PR record CRUD is on the orchestration server's REST surface for the SPA; engineer-initiated PR record creation goes through the role runtime's git/PR handling in step 09. If a future tool is needed (e.g., `create_pr` from inside the engineer subprocess), it lands as an additive change to the `mcp-engineer-surface` spec — and gets called out in `design.md` (Open Question 2).
  - **No WS streaming tools.** MCP is request/response; live updates flow over the orchestration server's `/events` WebSocket to the SPA, not to the engineer subprocess.
  - **No auth, no TLS, no signature on the agent header.** The MCP process and the orchestration server both run on `127.0.0.1`; the trust model from step 04 (local-only, role headers trusted) extends to MCP verbatim. Documented as a risk in `design.md`; future auth slots in *in front* of the role-identity middleware on the orchestration server, not inside the MCP layer.
  - **No persistent MCP-server state.** Each role-runtime cycle (per the §6 fresh-session rule) spawns a new MCP-server process with no shared memory across cycles; the only state the process holds is its three boot-time CLI arguments.
  - **No locking / leasing on tickets.** Two engineers in MVP could in principle race on the same ticket; for prototype we accept the orchestration server's request-level guards (status-graph + role-owner + storage atomicity) as sufficient. Step 26 introduces multi-engineer; this step's `design.md` notes what survives that change unmodified (the answer: nearly everything; the only seam that grows is "the role runtime hands the right `--agent` to the right MCP-server process").
  - **No prompt embedding work.** The engineer's prompt lands in step 09. This step ships only the verbs the prompt will name.

## Spec references

- §2#1 — Environment as communication bus. The MCP layer is the engineer's port to that bus.
- §2#3 — Status drives behaviour. `transition_ticket_status` is the engineer's only way to flip a status; the orchestration server enforces the §4.1 graph and the §4.2 owning-role rule.
- §4.2 — Owning-role rule. The MCP server stamps `X-Keni-Role: engineer`; the orchestration server's existing role guard enforces the rule.
- §5.3 — `.keni/` write boundary. The MCP server is the engineer's *only* legitimate write surface beyond their own workspace clone. No tool reads or writes `.keni/` directly; every write goes through the orchestration server.
- §5.4 — "All other reads and writes happen through MCP tools exposed by Keni." This step is the literal implementation of that sentence for engineers.
- §6.4 — Subprocess agnosticism. The MCP surface is what makes "Claude Code today, Cursor agent tomorrow" safe.
- §8 — Prototype "Included" list specifies "MCP surface for: tickets, activity log, workspace path." This step ships exactly that, and nothing more.
- §11#3 — Prompts as code. The engineer prompt (step 09) names the seven tools defined here; the prompt is code, not a file under `.keni/`.
