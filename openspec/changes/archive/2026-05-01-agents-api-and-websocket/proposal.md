## Why

Step 04 turned the file-backed storage from steps 01–03 into a proper local HTTP service: tickets, PRs, and the activity log are each writable through a role-guarded REST surface, status transitions are funnelled through the `§4.2` owning-role rule, and every response carries a stable envelope and `project_id`. What is missing — and what every later piece of Keni's UI is structurally blocked on — is the **reactive seam** between that server and the dashboard the user actually looks at: the SPA cannot render the spec's §7.2 "live as agents move tickets" promise off REST polling alone, the user cannot satisfy §6.1's "pause or resume any individual agent from the UI" affordance because there is no endpoint behind it, and the agent roster (left-hand region of the dashboard) has no way to report each agent's `status` / `last_activity` / `last_active_at` / `paused` flags. Step 05 closes that gap. It adds the `/agents` REST endpoints (list / pause / resume), wires a topic-style WebSocket event stream that broadcasts the writes the existing REST endpoints already perform (`ticket.*`, `pr.*`, `activity.appended`, `agent.state_changed`), and stands up the in-process plumbing — an event bus, a runtime agent-state tracker — that turns "the server processed a write" into "every connected SPA / role-runtime / curl-`websocat` consumer learns about it within milliseconds." It is also the foundation for step 08's scheduler (which will read the `paused` flag this step writes) and for step 12's optimistic-update layer in the SPA (which will reconcile against these events).

## What Changes

- Add the **`/agents` REST surface** to the existing Hono app:
  - `GET /agents` — returns `{ data: AgentResponse[], project_id }` where each row is `{ id, role, status: "idle" | "running", last_activity: string | null, last_active_at: string | null, paused: boolean }`. The roster source-of-truth is the `agents` array of the project's `.keni/project.yaml` (read at bootstrap from `ConfigStore`); the runtime fields (`status`, `last_activity`, `last_active_at`, `paused`) come from a new in-process `AgentRuntimeStateStore` populated by the activity-log feed and the pause/resume endpoints. Roles (`engineer` / `qa` / `po` / `writer`) are exposed verbatim. Unknown agent ids in `project.yaml` produce a default-paused row (`status: "idle"`, `paused: false`, last fields `null`).
  - `POST /agents/:id/pause` and `POST /agents/:id/resume` — flip the `paused` flag for the named agent and return `200 { data: AgentResponse, project_id }` with the post-mutation row. Both endpoints are idempotent (pausing a paused agent is a no-op success). Calls to an unknown agent id return `404 store_not_found`. The role guard accepts `X-Keni-Role: user` for both endpoints (the user is the source of pause/resume affordance per §6.1); other roles are rejected with `403 role_not_owner`. Step 08's scheduler will consume the resulting `paused` flag; this step does not yet wire the scheduler — it ships the API contract and the in-memory state machine the scheduler will read.
- Add the **WebSocket event stream** at `GET /events` (single topic-style channel; chosen for simplicity and forward-compatibility per `design.md` Decision 2):
  - Hono's `@hono/hono/deno` upgrade helper hosts a long-lived WS connection on the same port the REST surface listens on, so a single `keni start` (step 13) exposes both. The handshake reads `X-Keni-Role` from the upgrade request (browsers can't set arbitrary headers on `WebSocket(...)`, so the role is also accepted via the `?role=` query parameter — documented as the SPA-friendly path). Missing or unknown role rejects with status 401 / 1008 close code, mirroring the REST role guard's failure mode.
  - The connection multiplexes **six event types**, each carrying `{ id, event, project_id, timestamp, payload }` where `id` is a uuidv7 for client dedupe / future replay, `event` is one of `ticket.created`, `ticket.updated`, `pr.created`, `pr.updated`, `activity.appended`, `agent.state_changed`, and `payload` is a *minimal reference* (`{ ticket_id, status }`, `{ pr_id, status }`, `{ entry_id, agent, role, event }`, `{ agent_id, paused, status }`) — clients refetch full records via REST. This keeps event payloads small, decouples them from the storage record shape, and avoids racy "what was the field value at emit-time" semantics.
  - **Heartbeat / liveness** — the server sends a WS `ping` frame every 25 s (configurable in `design.md`); the client must respond with a `pong` within the same window. Connections that miss two consecutive heartbeats are closed with code `1011`. The SPA reconnects automatically.
  - **Reconnect semantics** — the prototype tier is "client refetches via REST on reconnect": no replay, no buffered backlog. The wire shape carries `id` so a future `?since=<event-id>` query-string replay is purely additive (no breaking change). The capability spec documents both the current tier and the additive seam.
- Add a server-side **event emitter** (`EventBus`) and **agent runtime-state store** (`AgentRuntimeStateStore`):
  - `EventBus` is a typed in-process pub/sub: `emit(event)` / `subscribe(handler): unsubscribe`. The new `/events` WS handler subscribes on connection and unsubscribes on close. Existing route handlers in `routes/tickets.ts`, `routes/prs.ts`, and `routes/activity.ts` gain a single `bus.emit(...)` call after each successful mutation; the emit is fire-and-forget (the bus catches subscriber errors so a failing WS client cannot poison the request handler). Pause/resume in the new agents endpoint emits `agent.state_changed`.
  - `AgentRuntimeStateStore` (in-memory, prototype-tier) holds the roster's runtime state keyed by `agent_id`: `{ paused, status, last_activity, last_active_at }`. It exposes `list()`, `read(id)`, `setPaused(id, value)`, and `applyActivityEvent(entry)` — both mutators return `{ state, changed }` so callers can debounce. The runtime-state update is **inline in the route handlers** that own the relevant write: `POST /activity` calls `applyActivityEvent(entry)` directly after the storage append (no bus subscriber, no refetch — the entry is already in scope), and `POST /agents/:id/pause` / `resume` call `setPaused(id, value)`. Each handler emits `agent.state_changed` on the bus *only when* the returned `changed` is `true` (debounce). The bus is purely fan-out for the WS endpoint; cross-cutting subscribers can be added additively later. The in-memory choice is documented in `design.md` (Decision 4) along with the seam for a future on-disk adapter (the interface stays stable; only the constructor argument changes when step 08 or a later persistence step lands).
- Define **wire shapes** in `@keni/shared/wire/`:
  - `agents.ts` — `AgentStatus` (`"idle" | "running"`), `AgentResponse`, `AgentListResponse`, `AgentEnvelope`, plus the runtime helpers (`AGENT_STATUSES` tuple, `isAgentStatus` type-guard).
  - `events.ts` — `EventName` (the six-name union), `EventEnvelope` (the WS frame), and one type per payload (`TicketCreatedPayload`, `TicketUpdatedPayload`, `PRCreatedPayload`, `PRUpdatedPayload`, `ActivityAppendedPayload`, `AgentStateChangedPayload`). The discriminated union `EventFrame = TicketCreatedFrame | TicketUpdatedFrame | …` is exposed for typed client consumption.
  - The runtime-helper / type split mirrors step 04: `@keni/shared/wire/` is types-only plus tiny runtime tuples for non-zod consumers; zod schemas live in `@keni/server/wire/`.
- Define **zod schemas** in `packages/server/src/wire/`:
  - `agents.ts` — `AgentStatusSchema`, `AgentResponseSchema`. (No request schemas needed: `GET /agents` has no body, and the pause / resume endpoints have empty bodies.)
  - `events.ts` — `EventEnvelopeSchema` plus the six payload schemas. The schema is used in *tests* (asserting the shape of every emitted frame round-trips) and as an outbound contract; the handler does not parse anything inbound on the WS channel today (the prototype connection is server-push-only — see `design.md` Decision 5).
  - Each schema is annotated with `z.ZodType<SharedType>` so the drift detector that step 04 introduced keeps catching misalignment at type-check time.
- Wire the **event emit seam** into existing routes:
  - `POST /tickets` → emit `ticket.created`. `PATCH /tickets/:id` and `POST /tickets/:id/transition` → emit `ticket.updated`. The PR and activity equivalents follow the same shape. The emit happens *after* the storage write returns and *before* `c.json(...)` so a successful HTTP response and a published event are paired (a bus subscriber that throws does not affect the response — the bus swallows and logs).
  - `POST /agents/:id/pause` and `POST /agents/:id/resume` → emit `agent.state_changed` with the post-mutation `paused` and `status`.
- Update `createServer` (composition root) to:
  1. accept the new `eventBus` and `agentRuntimeStateStore` in `ServerDeps` (additive — existing tests that build their own deps gain two simple in-memory instances), 2. mount the `agents` route group, 3. mount the `events` WS route group, 4. install the bus subscriber that updates the agent runtime state on each `activity.appended`. The middleware order is unchanged (`requestId → requestLog → roleIdentity → routes`); the WS upgrade handler runs after the same middleware stack so handshake failures still emit a request-log line.
- Update `runServer` to:
  1. read the project config once at bootstrap (already happens), 2. seed the `AgentRuntimeStateStore` from the `agents` array (each agent starts `paused: false`, `status: "idle"`, last fields `null`), 3. instantiate the `EventBus`, 4. pass both into `createServer`. No new CLI flags.
- Add **end-to-end integration tests**:
  - `routes/agents_test.ts` — list returns the configured roster; missing-id pause/resume → 404; engineer/qa/po pause-attempt → 403; user pause → 200 with `paused: true`; idempotent pause then resume; the post-mutation row carries the right `status`.
  - `routes/events_test.ts` — connect via `app.fetch(new Request(..., { upgrade: "websocket" }))` (Hono's documented test pattern, no port binding); `POST /tickets` produces a `ticket.created` frame within ms; `POST /tickets/:id/transition` produces a `ticket.updated` frame; `POST /activity` produces both `activity.appended` *and* an `agent.state_changed` if the entry's agent is in the roster and the event is a session marker; missing role on the WS upgrade → 401 close; heartbeat ping → pong round-trip; abort / reconnect leaves the server emitting cleanly to remaining subscribers.
  - `eventBus_test.ts` — emit / subscribe / unsubscribe; subscriber error does not propagate to the emit caller; multiple subscribers fan out; ordering is FIFO per subscriber.
  - `agentState_test.ts` — `applyActivityEvent` flips status / last fields per the rules; pause / resume idempotent; unknown id throws `StoreNotFoundError`.
  - Wire-schema tests — both new shape modules under `wire/`, with the same `z.ZodType<SharedType>` drift assertion the existing wire tests use.
- Extend the **`orchestration-server` capability spec** (delta) with new requirements for: the agents endpoint surface, the WS endpoint and event taxonomy, the in-memory `AgentRuntimeStateStore` contract, the event-emit seam in existing routes, and the reconnect / heartbeat / role-guard semantics on the WS channel. The middleware-order requirement is preserved verbatim (no change). The error-envelope requirement gains one new code (`agent_paused` is **not** in this step — it is reserved for step 08; the only new code is `unsupported_event_type`, used when a future client sends an unrecognised inbound message and the prototype refuses to parse it). The trust-model requirement is extended to name the `?role=` query-parameter as the WS-handshake equivalent of `X-Keni-Role`.
- Update the **root `README.md`** "Run the orchestration server" subsection with: the new `curl -H "X-Keni-Role: user" http://127.0.0.1:<port>/agents` example, a one-line `websocat -H 'X-Keni-Role: user' ws://127.0.0.1:<port>/events` example, and a note that pause / resume currently flip a flag the scheduler does not yet read (forward-reference to step 08).

## Capabilities

### New Capabilities

None. This step is structurally additive to the existing `orchestration-server` capability — every new endpoint, WS upgrade, and event lives on the same Hono app and is governed by the same middleware stack and trust model. Splitting the agents endpoint or the event stream into a separate capability would obscure that they share role-identity, error envelope, request-id propagation, and request-log shape with the rest of the HTTP surface. Future capabilities (`mcp-engineer-surface` in step 06, `chat-api` in step 15) will warrant their own spec files because they introduce different transports / role models.

### Modified Capabilities

- `orchestration-server`: add requirements for the `/agents` endpoint surface (list / pause / resume), the `/events` WebSocket stream (handshake, role guard, heartbeat, reconnect tier), the six-event taxonomy and minimal-reference payload shapes, the `EventBus` typed pub/sub contract, the in-memory `AgentRuntimeStateStore` contract (and the documented seam for future persistence), and the event-emit seam wired into the existing ticket / PR / activity route handlers. No existing requirement in the capability is changed or relaxed; the middleware order, error envelope, role-identity contract, project-id resolution rule, status-graph encoding, and trust model from step 04 are all preserved.
