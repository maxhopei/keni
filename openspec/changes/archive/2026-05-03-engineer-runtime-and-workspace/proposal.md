## Why

Steps 07 (role-runtime-common) and 08 (cron-scheduler-with-pause) gave Keni a deterministic spine that runs *any* role, but no role yet plugs into it. Without an engineer specialisation, the prototype's status board cannot move past `open` — there is nothing on disk to provision a code workspace, no bundled prompt to teach a coding agent the lifecycle (`open → in_progress → ready_for_review → in_review → has_comments|approved → merged → ready_for_test`), no precheck that knows which ticket to pick, and no docker-compose hook for integration tests. This change closes that gap so the prototype loop **user → ticket → engineer → PR → merge → ready_for_test** is end-to-end runnable with a single default engineer (`alice`).

## What Changes

- **New `@keni/role-runtimes` engineer module** under `packages/role-runtimes/src/engineer/` that exports `createEngineerRunner(deps, opts): AgentRunner` (compatible with the scheduler's registry from step 08), comprising:
  - An **engineer precheck** that calls `GET /tickets?status=open,test_failed,has_comments` against the orchestration server, applies the in-flight-ticket guard, and returns `{ kind: "skip", reason: "no_ticket_to_pick_up" }` or `{ kind: "proceed", roleContext: { summary: "ticket-NNNN" } }`. Spends zero subprocess tokens when there is nothing to pick up.
  - A **prompt resolver** that returns the bundled `BundledPrompt = { name: "engineer", body: <TS string constant> }`.
  - A **`workspacePath` resolver** keyed on `(projectId, agentId)` that yields `~/.keni/workspaces/<project-id>/<agent-id>/`, threaded through the role runtime as `RoleCycleParams.workspacePath` and exposed to the subprocess via the existing `KENI_MCP_WORKSPACE` env var and the MCP `get_workspace_path` tool from step 06.
- **New `WorkspaceProvisioner` interface and `GitWorkspaceProvisioner` default** under `packages/role-runtimes/src/engineer/workspace/` that owns the per-agent clone lifecycle:
  - `ensureProvisioned(projectId, agentId, projectRepoPath)` — creates `~/.keni/workspaces/<project-id>/<agent-id>/` with a git sparse-clone of the project repo (origin = on-disk project folder, sparse pattern excludes `.keni/` per spec.md §5.3), pulls `main` to the current tip, sets a deterministic per-workspace identity (no global `git config` writes), and is idempotent on subsequent calls.
  - `discardProvisioned(projectId, agentId)` — removes the workspace tree on engineer removal.
  - `pullMain(projectId, agentId)` — invoked at the start of each cycle to bring `main` up to date.
- **New bundled engineer prompt v1** as a TypeScript string constant under `packages/role-runtimes/src/engineer/prompts/engineer.ts`, opinionated for TS/Deno/React projects, teaching the loop documented in `spec.md` §3 (Engineer) and §4.1: pick top-of-queue ticket via MCP, plan, code in workspace, push branch `ticket-{id}`, submit PR via REST/MCP, self-review in a separate cycle (next tick), fix comments, merge, and call `transition_ticket_status` to `ready_for_test`. The prompt also documents the one-line summary contract from `spec.md` §6.3 and the `KENI_MCP_WORKSPACE` env contract.
- **New docker-compose integration-test hook** documented in the engineer prompt (the engineer invokes `docker-compose -f <project>/docker-compose.yml ...` from inside its own workspace; output streams to the activity log via the role runtime's stdout-line capture). No new package surface — just a documented contract that `<project>/docker-compose.yml` is the project-level integration-test entry point.
- **New merge-to-`main` mechanism** for the local-only prototype: a server-side `POST /prs/:id/merge` endpoint that accepts a PR id from the engineer's MCP layer and performs `git merge --ff-only` of the engineer's pushed branch (in the project repo on disk) onto `main`. Non-fast-forward attempts return `409 merge_conflict` and the engineer's prompt teaches the rebase-and-retry path. (Distinct from `transition_ticket_status` to `merged` — the engineer first calls `merge_pr`, then transitions the ticket.)
- **Wiring in `runServer`**: at bootstrap, after the scheduler is constructed, `runServer` instantiates the `GitWorkspaceProvisioner`, calls `ensureProvisioned(...)` for every engineer in the project's roster (default: `alice`), instantiates `createEngineerRunner(...)`, and registers it on the scheduler via `scheduler.registerRunner(runner)`. On `runServer` shutdown the provisioner is *not* discarded (workspaces persist across server restarts); only the engineer's removal from the roster discards them.

## Capabilities

### New Capabilities

- `engineer-runtime`: The engineer specialisation that plugs into the step-08 scheduler — `createEngineerRunner` factory, engineer precheck contract, workspace provisioner contract (`ensureProvisioned` / `discardProvisioned` / `pullMain`), sparse-checkout rules excluding `.keni/`, `ticket-{id}` branch convention, the `POST /prs/:id/merge` server-side fast-forward mechanism, the docker-compose integration-test contract, and `runServer` boot-time wiring (workspace provisioning + runner registration).
- `engineer-prompt`: The bundled engineer prompt v1 — its required sections (role identity, MCP tool inventory, ticket-pickup loop, branch / PR / merge workflow, self-review-in-fresh-session contract, integration-test invocation, summary-line contract), invariants (TS string constant compiled into the binary, never loaded from a user-editable file per `spec.md` §11#3), and `/opsx:verify` test fixtures that pin the prompt's expected behaviour.

### Modified Capabilities

- `orchestration-server`: Adds a new requirement that `runServer` wires up the built-in engineer role runner at bootstrap (after scheduler construction, before `Deno.serve` accepts connections), and adds a new `POST /prs/:id/merge` endpoint with engineer-only role guard, fast-forward-only semantics, and a `409 merge_conflict` response code added to the closed `ErrorCode` enum.

## Impact

- **New package paths**: `packages/role-runtimes/src/engineer/{runner,workspace,prompts}/...` plus tests; `packages/role-runtimes/src/main.ts` re-exports `createEngineerRunner` and `GitWorkspaceProvisioner`.
- **Modified package paths**: `packages/server/src/runServer.ts` (boot-time engineer wiring); `packages/server/src/routes/prs.ts` (new `POST /prs/:id/merge`); `packages/server/src/errors.ts` (new `merge_conflict` error code); `packages/server/src/wire/` (new merge-PR request/response types).
- **No new third-party dependencies**: every primitive is a Deno built-in (`Deno.Command` for `git` / `docker-compose`), an `@std/*` module already in `deno.json` (`@std/path`, `@std/fs` for workspace tree management), or an existing `@keni/*` workspace package.
- **Filesystem footprint**: per-engineer workspace tree at `~/.keni/workspaces/<project-id>/<agent-id>/` (cloned at first server boot for each engineer in the roster; default `~/.keni/workspaces/<project-id>/alice/`). The project root's git repo gains one merge commit per `POST /prs/:id/merge` call.
- **CI / smoke test**: a new integration test under `packages/role-runtimes/src/engineer/integration_test.ts` drives a no-op-but-realistic engineer cycle end-to-end against a seeded project (one open ticket, a workspace cloned to a temp `~/.keni/`, a fake coding-agent fixture that emits a deterministic plan + summary), and verifies (a) the precheck picks the open ticket, (b) the workspace is provisioned with `.keni/` excluded, (c) the activity log gains the documented entries, and (d) `merge_pr` against the engineer's pushed branch fast-forwards `main`.
- **Out of scope** (deferred to later steps): multi-engineer parallelism (step 26), self-review automation (the prompt covers it conceptually; the prototype runs single-engineer and self-review happens on the engineer's *next* tick), manual override flow (step 25), engineer-specific event-driven triggers (post-MVP).
